# PinkSurvivor 결정 로그

최종 수정: 2026-04-04

## 사용 규칙

- 현재 구현을 재현하는 데 영향을 주는 설계 결정만 기록한다.
- 코드와 자산이 바뀌면 이 문서도 같이 갱신한다.
- 일시적인 아이디어가 아니라, 재현 기준으로 유지할 값만 남긴다.

## 결정 사항

### 2026-04-04 | 문서 기준 재설정 | `TopDownTestOne` 문서를 폐기하고 `PinkSurvivor` 기준으로 재작성

- 기존 문서는 다른 프로젝트명, 다른 엔진 버전, 다른 구조를 가리키고 있었다.
- 재현 문서는 현재 저장소의 실제 코드와 자산을 기준으로 다시 고정한다.

### 2026-04-04 | 복원 기준 분리 | `같은 플레이`와 `같은 바이너리`를 구분

- 문서의 1차 목표는 `Gameplay-equivalent` 복원이다.
- 자산 이름, 경로, 부모 클래스까지 맞추는 `Asset-path-equivalent`를 2차 목표로 둔다.
- `Binary-identical`은 원본 아트와 생존 `uasset`가 없으면 보장하지 않는다.

### 2026-04-04 | 장르 방향 | 3D 뱀서 루프 유지

- 카메라와 표현만 3D로 옮기고, 핵심 루프는 `이동 -> 자동 공격 -> 적 처치 -> 젬, 코인 획득 -> 성장` 구조를 유지한다.

### 2026-04-04 | 런타임 클래스 연결 | 실제 플레이에는 블루프린트 자산을 우선 사용

- 핵심 로직은 C++에 둔다.
- 실제 플레이 자산 연결은 `BP_SVGameMode`, `BP_Player`, `BP_Enemy`, `WBP_HUD`를 기준으로 맞춘다.
- 이렇게 해야 메시, 입력 자산, 위젯 클래스 연결까지 동일하게 재현된다.

### 2026-04-04 | HUD 구조 | HUD 연출은 `AHUD + Widget BP` 혼합 유지

- `APSVHUD`가 위젯 생성, 캐시 전달, 룰렛 텍스트 직접 그리기를 담당한다.
- `WBP_PlayerHUD`, `WBP_Gameover`는 각각 플레이 HUD와 게임 오버 UI를 담당한다.
- `WBP_HUD`는 이름과 달리 `UUserWidget`이 아니라 `APSVHUD` 기반 블루프린트다.
- 룰렛 표시 텍스트는 `WBP_PlayerHUD` 내부가 아니라 `APSVHUD::DrawHUD()` 캔버스 드로우로 유지한다.

### 2026-04-04 | 플레이어 무기 구조 | 수동 발사 대신 전방 자동 발사 유지

- 현재 기본 무기는 `UPSVAutoFireComponent`가 0.5초마다 발사체를 전방으로 생성하는 구조다.
- 별도 조준 루프 없이 움직임과 생존에 집중하는 감각을 유지한다.
- 디버그, 치트 성격의 8방향 발사는 옵션으로만 유지한다.

### 2026-04-04 | 적 AI 구조 | Behavior Tree 없이 단순 추적, 근접 공격 유지

- 적은 Tick에서 플레이어를 향해 이동한다.
- 근접 거리 진입 시 타이머 기반 공격 루프를 시작한다.
- 현재 구현은 복잡한 AI보다 반복 전투 루프 재현을 우선한다.

### 2026-04-04 | 성장, 경제 구조 | 경험치는 런 단위, 골드는 영구 저장

- 경험치와 레벨은 사망 시 초기화한다.
- 골드는 `UPSVGameInstance + UPSVSaveGame` 조합으로 `PlayerProgress` 슬롯에 저장한다.
- 이 분리는 뱀서식 런 성장과 메타 경제를 동시에 재현하기 위한 기준이다.

### 2026-04-04 | 상자 처리 | 보상보다 연출과 입력 흐름을 먼저 고정

- 상자는 현재 실제 업그레이드를 주지 않는다.
- 대신 `게임 일시정지 -> 후보 텍스트 순환 -> 확인 입력 대기 -> 게임 재개` 흐름을 먼저 고정했다.
- `ConfirmAction`이 일시정지 중에도 동작해야 하는 이유가 여기에 있다.

### 2026-04-04 | 입력 복원 규칙 | Enhanced Input 3자산 구조 유지

- 입력은 `IA_Move`, `IA_Confirm`, `IMC_Default` 3자산 구조를 유지한다.
- `IA_Move`는 `Axis2D`, `IA_Confirm`은 `Boolean`이다.
- `IMC_Default`는 `W/A/S/D`와 `SpaceBar` 구성이 가장 유력하고, 코드 쪽 우선순위는 `0`이다.

### 2026-04-04 | 스폰 시스템 | 고정 스폰 스크립트와 스칼라 기반 반복 검증 유지

- GameMode는 4개의 시간 오프셋 스폰 명령을 내장한다.
- `SpawnCountScalar = 0.25`로 실제 테스트 물량을 줄여 빠르게 검증한다.
- 동일 구현을 재현할 때는 이 스칼라와 명령 조합을 같이 맞춘다.

### 2026-04-04 | 자산 재사용 | 캐릭터 외형은 Hoshino, 픽업, 탄환은 단순 메시 유지

- 플레이어와 적은 모두 `Hoshino` 계열 메시를 기반으로 한다.
- 젬, 코인, 탄환은 기본 도형 + 단색 머티리얼로 구성한다.
- 상자는 전용 `Chest` 메시를 사용한다.

### 2026-04-04 | 삭제 후 복원 워크플로우 | `Saved/Autosaves`를 실복원 자원으로 취급

- `Content` 삭제 후 exact 복원 가능성이 가장 높은 것은 `Saved/Autosaves`에 남은 자산들이다.
- 따라서 문서는 `Source/Config > Saved/Autosaves > Content 문자열 분석 > Docs` 순으로 복원 우선순위를 둔다.

### 2026-04-04 | clean rebuild 범위 | `BP_EnemySpawner`는 현재 필수 대상에서 제외

- 저장소 안에 `PSVEnemySpawner` C++ 클래스가 없고, 맵 배치 흔적도 없다.
- 따라서 현재 clean rebuild 목표는 `BP_EnemySpawner` 없이도 충족된다.
