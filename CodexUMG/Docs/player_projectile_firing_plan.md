# 플레이어 투사체 발사 구현 계획

## 문서 목적
- 본 문서는 플레이어가 투사체를 발사하는 기능을 설계하고 지시하기 위한 상위 계획 문서다.
- 프로젝트의 현재 구현 상태를 미리 확정하지 않고, 어떤 구조에도 적용 가능하도록 유연한 기준만 정의한다.
- 가장 중요한 우선순위는 리드타임 단축이며, 병렬 처리 가능한 작업은 가능한 한 에이전트 단위로 분리한다.

## 핵심 원칙
- 투사체 클래스는 실제 존재하는 Blueprint Asset로 만든다.
- 투사체 클래스 참조는 `DA_` 계열 Data Asset에 보관한다.
- 해당 Data Asset은 `GameInstance`에서 연결하고 제공한다.
- 플레이어 관련 클래스는 `GameInstance` 또는 그를 통해 노출된 접근 지점을 사용해 참조를 얻는다.
- 플레이어 클래스가 특정 투사체 Blueprint를 직접 들고 있거나 직접 연결하지 않는다.
- 투사체 Blueprint는 기본적으로 Static Mesh를 포함해야 하며, 스폰 직후 사용자가 월드에서 확인할 수 있어야 한다.
- 투사체의 Static Mesh는 이후 `BP_Projectile_*` 에디터 설정에서 다른 메시로 교체 가능한 구조여야 한다.
- 기능 구현 에이전트는 자신의 산출물을 스스로 평가하지 않는다.
- 산출물 평가는 반드시 별도의 평가 전용 에이전트가 수행한다.

## 목표 상태
- 플레이어 측에서 발사 요청이 발생하면, 플레이어 관련 로직은 직접 투사체 Blueprint를 알지 않아도 발사 가능해야 한다.
- 발사에 필요한 투사체 종류, 기본 설정, 확장 포인트는 Data Asset 기반으로 바꿀 수 있어야 한다.
- 입력 처리 방식은 기존 구조를 우선 재사용하고, 불가할 때만 최소 범위의 입력 자산을 추가한다.
- 이후 단일 발사, 연사, 차지샷, 속성 탄환, 무기별 다른 투사체 등으로 확장 가능해야 한다.

## 입력 처리 분기 원칙

### 1. 기존 발사 처리 재사용 우선
- 기존 플레이어에 이미 연결된 발사 관련 처리 흐름이 있다면 그 방식을 우선 따른다.
- 이 경우 새 입력 체계를 억지로 도입하지 않고, 기존 발사 진입점에서 투사체 데이터 참조 구조만 정리하는 방향을 우선 검토한다.

### 2. 기존 발사 처리 없음, 기존 IMC_ 존재
- 플레이어에 연결된 기존 발사 관련 처리가 없다면, 플레이어가 사용 중인 기존 `IMC_`에 발사용 `IA_`를 추가해 연결한다.
- 새 입력 자산은 기존 입력 체계 안에서 최소 범위로 추가하며, 별도 독립 입력 구조를 병행 생성하지 않는다.
- 명명은 프로젝트 규칙을 따르되, 문서 기준으로는 `IA_Fire` 또는 동등한 의미의 이름을 권장한다.

### 3. IMC_ 연결 흔적 자체가 없음
- 플레이어가 어떤 `IMC_`에도 연결되어 있지 않은 상태라면, 본 작업은 시작하지 않는다.
- 이 경우 먼저 사용자에게 최소 수준의 Enhanced Input 구현을 요구한다.
- 최소 요구 범위는 다음 정도로 본다.
  - 플레이어 또는 그에 준하는 입력 수신 주체에 `IMC_` 연결 경로가 존재할 것
  - 하나 이상의 `IA_`가 실제 입력 이벤트로 연결될 것
  - 이후 발사 입력을 추가할 수 있는 기본 입력 파이프라인이 확인될 것

## 권장 구조

### 1. 자산 구조
- `BP_Projectile_*`
  - 실제 스폰 가능한 투사체 Blueprint Asset
  - 기본적으로 Static Mesh Component를 포함
  - 스폰 시 월드에서 식별 가능한 기본 메시를 가짐
  - 이후 에디터에서 메시를 교체해도 발사 구조 자체는 유지됨
- `DA_Projectile_*` 또는 이를 포함하는 상위 `DA_`
  - 투사체 클래스 참조
  - 필요 시 속도, 수명, 이펙트, 충돌 정책, 데미지 정책 등 확장용 필드 포함
- `GameInstance`
  - 어떤 `DA_`를 사용할지 연결
  - 플레이어 관련 시스템이 접근 가능한 진입점 제공
- `IMC_*`
  - 기존 입력 체계를 재사용하는 기본 입력 매핑 컨텍스트
- `IA_Fire` 또는 동등한 발사 액션 자산
  - 기존 발사 처리 흐름이 없을 때만 추가 후보가 됨

### 2. 참조 흐름
1. 기존 발사 처리 또는 입력 액션을 통해 발사 요청 발생
2. 플레이어 관련 클래스는 직접 투사체 BP를 참조하지 않고 발사 의도만 전달
3. 요청 처리 지점은 `GameInstance` 경유로 `DA_`에서 필요한 투사체 정의를 조회
4. 조회된 Blueprint Class를 기반으로 투사체 스폰
5. 스폰 후 초기화 파라미터와 기본 가시성 상태 적용

### 3. 책임 분리
- 플레이어 클래스
  - 발사 의도 전달
  - 기존 발사 처리 재사용 여부 판단의 진입점 보유 가능
  - 투사체 클래스 직접 참조는 금지
- 입력 계층
  - 기존 발사 처리 재사용 또는 기존 `IMC_`에 `IA_` 추가
  - 입력 체계가 없으면 작업 중단 조건을 만족했는지 확인
- 데이터 접근 계층
  - `GameInstance` 경유로 현재 사용해야 할 `DA_` 제공
- 발사 처리 계층
  - 스폰 위치, 방향, 초기 속성 적용
- 투사체 자산 계층
  - 실제 비주얼, 충돌, 이동, 명중 후 처리

## 필수 요구사항
- 실제 스폰 가능한 `BP_Projectile_*` 자산이 존재해야 한다.
- `BP_Projectile_*`는 기본 Static Mesh를 포함해야 한다.
- 해당 Static Mesh는 스폰 즉시 플레이어가 식별 가능해야 한다.
- 메시 자산은 이후 `BP_Projectile_*` 에디터에서 교체 가능해야 한다.
- `DA_`가 투사체 Blueprint Class를 참조해야 한다.
- `GameInstance`가 해당 `DA_`를 연결하고 플레이어 관련 클래스에 접근 경로를 제공해야 한다.
- 플레이어 관련 클래스는 직접 투사체 Blueprint를 참조하지 않아야 한다.
- 입력 처리는 아래 우선순위를 따라야 한다.
  - 기존 발사 처리 재사용
  - 없으면 기존 `IMC_`에 `IA_` 추가
  - `IMC_` 자체가 없으면 작업 시작 금지 후 사용자에게 최소 Enhanced Input 구현 요청

## 구현 범위 제안

### 필수 범위
- 실제 투사체 Blueprint Asset 생성
- 투사체 Blueprint에 기본 Static Mesh 구성
- `DA_`에 투사체 클래스 참조 연결
- `GameInstance`에 `DA_` 연결
- 플레이어 관련 클래스가 `GameInstance` 경유로 데이터에 접근하도록 구조 정리
- 플레이어 클래스에서 직접 투사체 Blueprint를 물고 있는 경로 제거 또는 금지
- 입력 처리 분기 규칙 적용
- 기본 발사 흐름 1종 완성

### 선택 범위
- 다중 투사체 타입 전환
- 데이터 기반 발사 파라미터 확장
- 애니메이션, 이펙트, 사운드 연동
- 서버 권한, 예측, 재동기화 정책
- 디버그용 발사 로그 및 시각화

## 리드타임 단축을 위한 에이전트 운영 방식

### 기본 방침
- 병렬로 가능한 작업은 적극적으로 에이전트에 위임한다.
- 에이전트 간 의존성은 최소화하고, 먼저 인터페이스와 산출물 형식만 짧게 합의한 뒤 동시 작업한다.
- 각 에이전트는 자신의 담당 산출물만 책임지고, 최종 품질 판정은 하지 않는다.

### 권장 에이전트 분리

#### Agent A: 투사체 자산 담당
- 목표
  - `BP_Projectile_*` 생성 기준 정의
  - 기본 Static Mesh 포함 구조 정의
  - 에디터에서 메시 교체 가능한 구성 기준 정리
- 주요 산출물
  - 투사체 BP 명명 규칙
  - 기본 컴포넌트 구성안
  - 기본 메시 가시성 기준

#### Agent B: 데이터 자산 및 GameInstance 담당
- 목표
  - `DA_Projectile_*` 또는 상위 `DA_` 구조 정의
  - `GameInstance`가 `DA_`를 연결하고 노출하는 방식 정리
- 주요 산출물
  - Data Asset 필드 초안
  - `GameInstance -> DA_ -> Projectile Class` 접근 흐름
  - 초기화 시점, 접근 함수, 실패 처리 기준

#### Agent C: 입력 연동 담당
- 목표
  - 발사 입력 연동 규칙을 결정 트리 형태로 정리
- 주요 산출물
  - 기존 발사 처리 재사용 기준
  - 기존 `IMC_`에 `IA_`를 추가하는 최소 작업 기준
  - `IMC_` 흔적이 없을 경우 작업 중단 및 사용자 요청 문구 초안

#### Agent D: 플레이어 발사 요청 담당
- 목표
  - 플레이어 관련 클래스가 직접 투사체 BP를 들지 않도록 발사 요청 경로 정리
- 주요 산출물
  - 발사 요청 인터페이스 또는 호출 흐름
  - 플레이어 클래스 책임 최소화 방안

#### Agent E: 스폰 및 초기화 담당
- 목표
  - 실제 투사체 스폰 시 필요한 최소 절차 정리
- 주요 산출물
  - 스폰 위치, 방향, 소유자, instigator, 초기 파라미터 적용 규칙
  - 기본 메시가 사용자에게 보이는 상태인지 확인할 기준
  - 실패 시 안전한 폴백 기준

#### Agent F: 확장 포인트 담당
- 목표
  - 이후 기능 확대 시 데이터 구조가 깨지지 않도록 옵션 정리
- 주요 산출물
  - 속성탄, 무기별 탄, 차지샷, 다중 발사 대응 포인트
  - 네트워크, 이펙트, 사운드 확장 훅

#### Agent G: 문서 통합 담당
- 목표
  - 각 에이전트 산출물을 하나의 실행 가능한 작업 문서로 통합
- 주요 산출물
  - 최종 구현 체크리스트
  - 의존성 순서와 병렬 처리 구간 정리

#### Agent H: 평가 전용 담당
- 목표
  - 다른 에이전트의 산출물을 독립적으로 검토
- 주요 산출물
  - 구조 위반 여부 점검
  - 리드타임 관점에서 병렬화가 충분한지 평가
  - 플레이어 클래스 직접 참조 금지 원칙 준수 여부 확인
  - 입력 처리 분기 규칙 누락 여부 확인
  - Static Mesh 가시성 및 교체 가능성 요구사항 반영 여부 확인
- 주의
  - 구현 에이전트 본인이 자기 결과를 평가하지 않는다.
  - 평가 결과는 수정 요청의 입력으로만 사용하고, 평가 에이전트가 직접 구현까지 맡지는 않는다.

## 권장 작업 순서

### 1차 병렬 구간
- Agent A: 투사체 자산 구조 정의
- Agent B: `DA_` 및 `GameInstance` 연결 방식 정의
- Agent C: 입력 처리 분기 규칙 정의

### 2차 병렬 구간
- Agent D: 플레이어 발사 요청 구조 정의
- Agent E: 스폰 및 초기화 규칙 정리
- Agent F: 확장 포인트 정리

### 통합 구간
- Agent G: 전체 산출물 통합

### 평가 구간
- Agent H: 독립 평가 수행

## 최소 인터페이스 합의안
- 플레이어 관련 클래스는 "발사 요청이 발생했다" 수준만 전달한다.
- 입력 계층은 기존 발사 처리 재사용 여부를 먼저 판단한다.
- 기존 발사 처리 경로가 없으면 기존 `IMC_`에 발사용 `IA_`를 추가하는 방향으로만 확장한다.
- `IMC_` 연결 흔적이 없으면 구현을 중단하고 사용자에게 최소 Enhanced Input 구현을 먼저 요구한다.
- `GameInstance`는 현재 유효한 `DA_` 또는 그에 준하는 데이터 접근 지점을 제공한다.
- `DA_`는 실제 투사체 Blueprint Class를 반환할 수 있어야 한다.
- 발사 처리 계층은 반환된 클래스를 기반으로 스폰한다.
- 스폰된 투사체는 기본 Static Mesh를 통해 즉시 식별 가능해야 한다.

## 금지 사항
- 플레이어 클래스에 특정 `BP_Projectile_*`를 직접 변수로 박아두는 방식
- 플레이어 클래스가 `DA_` 없이 투사체 타입을 하드코딩하는 방식
- 기존 발사 처리 경로가 있는데도 별도 입력 체계를 중복 추가하는 방식
- 기존 `IMC_`가 있는데도 불필요하게 신규 `IMC_`를 병행 도입하는 방식
- `IMC_` 연결 흔적이 없는데도 임의 구현을 시작하는 방식
- 사용자가 확인할 수 없는 상태로 투사체를 스폰하는 방식
- 투사체 메시를 BP 에디터에서 교체하기 어렵게 고정하는 방식
- 구현 담당 에이전트가 자기 산출물의 최종 평가까지 겸하는 방식
- 병렬 가능한 작업을 불필요하게 한 에이전트에 몰아 리드타임을 늘리는 방식

## 체크리스트
- 실제 스폰 가능한 투사체 Blueprint Asset이 정의되었는가
- 투사체 Blueprint가 기본 Static Mesh를 포함하는가
- 스폰 직후 사용자가 투사체를 식별 가능한가
- 추후 `BP_Projectile_*` 에디터에서 메시를 교체 가능한가
- `DA_`가 투사체 Blueprint Class를 참조하는가
- `GameInstance`가 해당 `DA_`를 연결하고 접근 가능하게 하는가
- 플레이어 관련 클래스가 직접 투사체 Blueprint를 참조하지 않는가
- 기존 발사 처리 재사용 여부를 먼저 판단하도록 계획되었는가
- 기존 발사 처리 부재 시 기존 `IMC_`에 `IA_`를 추가하는 규칙이 반영되었는가
- `IMC_` 연결 흔적이 없을 경우 작업 중단 및 사용자 요청 규칙이 반영되었는가
- 발사 요청과 실제 스폰 책임이 분리되어 있는가
- 병렬 처리 가능한 항목이 에이전트로 분리되어 있는가
- 최종 평가는 별도 평가 에이전트가 수행하도록 계획되었는가

## 메모
- 실제 클래스명, 함수명, 자산명, 폴더 구조는 프로젝트 룰에 맞게 조정한다.
- 본 문서는 구체 구현서가 아니라 작업 지시와 구조 정렬을 위한 계획 문서다.
- 세부 설계는 이 문서를 기반으로 축약하거나 확장해도 되지만, 아래 원칙은 유지한다.
  - 투사체 참조는 `DA_`와 `GameInstance`를 경유한다.
  - 플레이어 클래스는 투사체 Blueprint를 직접 연결하지 않는다.
  - 입력 처리는 기존 구조 재사용을 우선한다.
  - 입력 기반이 없으면 먼저 최소 Enhanced Input 기반을 사용자에게 요구한다.
