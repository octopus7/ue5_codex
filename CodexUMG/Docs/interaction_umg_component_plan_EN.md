# Interaction UMG Component Implementation Plan

## Document Purpose
- Define a component structure that displays interaction availability as a screen-space UMG marker attached to a world target as the player gets closer.
- Treat it as a fixed rule that the real `Widget Blueprint` asset and the circular marker texture assets are generated by editor-module code, while generation is executed through a `Commandlet`.
- Let the interaction type be selectable on the component, but route the actual action-entry path only through a global subsystem.
- Plan the minimum vertical slice that is immediately verifiable, including test apple/strawberry Blueprints.
- Also define an operating model that splits parallelizable work by agent to reduce lead time.

## Target State
1. If `UCodexInteractionComponent` is attached to a world object, a screen-space UMG marker appears based on distance to the player.
2. The widget uses only three states: `Hidden`, `VisibleRange`, and `Interactable`.
3. On `Hidden -> VisibleRange`, the filled circle and outer ring fade in, and the ring starts larger and more transparent, then shrinks until it wraps the filled circle.
4. On `VisibleRange -> Hidden`, the same animation plays in the exact reverse direction.
5. When the state becomes `Interactable`, a white background box with black text appears to the right of the circle marker.
6. Only the single closest target can be in the `Interactable` state, so the white-box + black-text prompt is always shown only once.
7. When interaction input occurs, the component does not handle it directly and only forwards a request to the global subsystem.
8. The global subsystem prints screen debug strings only for major events such as interaction request, interaction end, and no candidate; focus start/end remain log-only.
9. Real `BP_Interactable_Apple` and `BP_Interactable_Strawberry` assets must exist, and both operate as the `Eat` interaction.
10. The related textures, widget asset, and test Blueprint assets must all be creatable through the editor-module + commandlet path.

## Core Principles
- The interaction component stores distance/display metadata, registers itself with the subsystem, and only reflects state.
- The decision of which target is closest is not made by each component independently. A global subsystem makes that decision centrally.
- The interaction prompt opens only on the single target chosen by the subsystem.
- The real `Widget Blueprint` asset and circular marker `Texture2D` assets must exist under project `Content`, and they must be produced through editor code + commandlet rather than manual creation.
- The filled circle and ring textures are generated by editor-module code that writes the pixel data directly instead of importing external source images.
- Both textures use white RGB only, and define the shape only through the alpha channel.
- Input expansion must first check whether an existing `Enhanced Input` implementation exists, then follow that structure exactly.
- If no existing `Enhanced Input` implementation exists, stop the interaction feature immediately and ask the user to implement a minimum `Enhanced Input` foundation first.
- For widget state-transition animation, prefer handling scale/opacity interpolation in a C++ base widget over assembling complex `WidgetAnimation` tracks inside the commandlet.
- The interaction type is chosen via a component `enum`, but the component does not directly execute gameplay such as `Eat`.
- Because the first implementation needs world-level state evaluation and ticking, prefer `UWorldSubsystem` as the global receiver.
- Keep configuration data and input-asset references aligned with the existing `GameInstance + DataAsset` pattern as much as possible.

## Recommended Runtime Structure

### 1. `UCodexInteractionComponent`
- Owned data
  - interaction type `ECodexInteractionType`
  - display text `FText`
  - visible distance
  - interactable distance
  - widget offset
  - reference to the generated `Widget Blueprint` class
- Responsibilities
  - register itself with the subsystem at BeginPlay
  - unregister itself from the subsystem at EndPlay
  - create and attach a `UWidgetComponent` at runtime to the owning actor
  - reflect the subsystem-computed state into the widget
  - forward input requests to the subsystem instead of handling them directly
- Non-responsibilities
  - judging which candidate is closest
  - executing the actual `Eat` gameplay
  - comparing priority against other candidates

### 2. `UCodexInteractionIndicatorWidget`
- This is the C++ base class of `WBP_InteractionIndicator`.
- `BindWidget` targets
  - `IMG_FilledCircle`
  - `IMG_OuterRing`
  - `BOR_PromptBackground`
  - `TXT_Prompt`
- Internal state
  - current state
  - target state
  - interpolation progress
- Responsibilities
  - visual state transitions among `Hidden`, `VisibleRange`, and `Interactable`
  - interpolate `RenderOpacity` and `RenderScale` for the two circular markers
  - handle text, visibility, and simple fade/slide interpolation for the prompt box
- Implementation rules
  - initialize to the default hidden state in `NativeConstruct`
  - handle transitions using `NativeTick` or timer-based interpolation
  - the `Hidden <-> VisibleRange` transition must be able to play in reverse with the same parameters
  - the `VisibleRange -> Interactable` transition keeps the circular marker and opens only the prompt box in addition

### 3. `UCodexInteractionSubsystem`
- A global receiver based on `UWorldSubsystem`.
- Responsibilities
  - maintain the list of registered `UCodexInteractionComponent`s
  - compute current candidate distances relative to the local player
  - choose the closest currently interactable candidate
  - distribute target widget states to each component
  - receive interaction requests and print debug output
  - call the target actor's `UCodexInteractionTarget` interface when needed
- Why this choice
  - it needs world-level ticking
  - the closest-target decision must be made centrally
  - debug output and focus state are tightly tied to world context
- Focus rules
  - `distance > visible distance` -> `Hidden`
  - `interactable distance < distance <= visible distance` -> `VisibleRange`
  - `distance <= interactable distance` -> included in the interactable candidate set
  - only the single closest candidate becomes `Interactable`
  - all other candidates remain `VisibleRange`
- Equal-distance stability rules
  - if the distance difference is very small, use a stable tie-break rule such as registration order or actor name
  - do not allow flicker where the chosen target flips every frame

### 4. `UCodexInteractionTarget` interface
- Purpose
  - provides a receiving point so the interaction component does not execute gameplay directly
  - provides the minimum path for test apple/strawberry Blueprints to react to `Eat`
- Recommended functions
  - `HandleInteractionRequested(const FCodexInteractionRequest&)`
  - `HandleInteractionEnded(const FCodexInteractionRequest&)`
- First-pass test behavior
  - debug output when a request is received
  - after a short delay, hide the actor or call `DestroyActor`

## Widget State Definitions

### 1. `Hidden`
- filled circle image `Opacity = 0`
- outer ring image `Opacity = 0`
- outer ring image `Scale > 1.0`
- prompt box hidden

### 2. `VisibleRange`
- the filled circle image fades in gradually
- the outer ring starts at a larger size and shrinks until it wraps the filled circle
- the outer ring remains more transparent than the filled circle
- the prompt box does not open

### 3. `Interactable`
- keeps the circular marker from `VisibleRange`
- shows a white background box + black text to the right of the circle marker
- uses the text owned by the component
- this state is allowed for only one closest target

## Marker Texture Spec

### Shared rules
- The textures are generated directly at the pixel level by editor-module code.
- `UCodexInteractionAssetBuildCommandlet` is responsible for the generation run.
- Both textures use pure white RGB `255,255,255`.
- Shape is created only through the alpha channel.
- Outside the shape, `Alpha = 0`.
- Inside the shape, `Alpha = 255` by default.
- If edge antialiasing is needed, interpolate alpha only within the 1-pixel boundary region, but keep the shape-defining radii below fixed.

### `T_InteractionFilledCircle`
- Usage
  - brush texture for `IMG_FilledCircle`
- Resolution
  - `64x64`
- Shape
  - radius `16` from center
  - inside radius `16` is the valid white + alpha region
  - outside radius `16` is fully transparent

### `T_InteractionOuterRing`
- Usage
  - brush texture for `IMG_OuterRing`
- Resolution
  - `64x64`
- Shape
  - outer radius `24` from center
  - ring thickness `4`
  - therefore the inner radius is `20`
  - only the `20 ~ 24` radius band is the valid white + alpha region
  - regions `< 20` and `> 24` are fully transparent

## Recommended Visual Parameters
- Filled circle
  - texture: `T_InteractionFilledCircle`
  - brush size: `48x48`
  - `Hidden -> VisibleRange`: `Opacity 0.0 -> 1.0`
- Outer ring
  - texture: `T_InteractionOuterRing`
  - brush size: `48x48`
  - `Hidden -> VisibleRange`: `Opacity 0.0 -> 0.6`
  - `Hidden -> VisibleRange`: `Scale 1.6 -> 1.0`
- Prompt box
  - `VisibleRange -> Interactable`: `Opacity 0.0 -> 1.0`
  - `VisibleRange -> Interactable`: X offset `+12 -> 0`
- Widget component
  - `DrawSize`: `280x72`
  - pivot: approximately `24 / 280` on X so the marker center aligns with the actor position
- Transition time
  - use a short default interpolation in the `0.15` to `0.25` second range
- Reverse transition
  - when moving away, interpolate the values above in reverse

## Input And Request Flow
1. First check whether the project already has an `Enhanced Input` implementation.
2. The minimum verification criteria are:
   - a binding path using `EnhancedInputComponent` must exist
   - a path that adds `IMC_` to `UEnhancedInputLocalPlayerSubsystem` must exist
   - the existing `IA_`, `IMC_`, and `DA_` reference structure must already be wired for real
3. If those conditions are met, any new input must extend that exact structure.
4. If no existing `Enhanced Input` implementation exists, stop the work here and ask the user to implement a minimum `Enhanced Input` foundation first.
5. Add an `InteractAction` reference to `UCodexTopDownInputConfigDataAsset`.
6. Create `IA_Interact` through the editor module + commandlet.
7. Add `IA_Interact` to the existing `IMC_TopDown`.
8. Fix the key mapping of `IA_Interact` to the `F` key.
9. Add interaction binding to [Source/CodexUMG/Private/CodexTopDownPlayerController.cpp](../Source/CodexUMG/Private/CodexTopDownPlayerController.cpp) in the same style as the existing binding pattern.
10. The player controller does not search for target actors directly. It only sends a "current interaction request" to the subsystem.
11. If there is a component currently in the `Interactable` state, the subsystem creates an `FCodexInteractionRequest`.
12. The subsystem prints screen debug strings only for interaction request/end and no-candidate cases.
13. The subsystem forwards follow-up calls only when the target actor implements the `UCodexInteractionTarget` interface.
14. The test apple/strawberry Blueprints receive that call and execute the `Eat` behavior.

  - load `BasicMap` and place or reposition the test apple/strawberry near the player start path

  - compile and save assets
- Execution prerequisites
  - run while no editor UI session is open
  - if a conflict risk appears, stop immediately and tell the user

### 3. `WBP_InteractionIndicator` creation method
- Create a real `Widget Blueprint` asset with `UWidgetBlueprintFactory`.
- Fix the base class to `UCodexInteractionIndicatorWidget`.
- Default widget tree
  - Root: `CanvasPanel`
  - Marker Group: `Overlay`
  - `IMG_FilledCircle`
  - `IMG_OuterRing`
  - `BOR_PromptBackground`
  - `TXT_Prompt`
- `IMG_FilledCircle` uses `T_InteractionFilledCircle` as its brush.
- `IMG_OuterRing` uses `T_InteractionOuterRing` as its brush.
- For `IMG_FilledCircle` and `IMG_OuterRing`, write `Brush.ImageSize = 48x48` directly when generating the asset.
- Do not rely only on calling `SetDesiredSizeOverride()`.
  - Reason: when a `Widget Blueprint` is created by editor code, the size may not be saved into the designer asset and `Image Size = 0,0` can remain.
- Place the prompt box to the right of the circular marker.
- The default visibility must be hidden.
- Finish widget-tree construction in code and handle animation through runtime C++ interpolation.

### 4. Texture generation method
- The editor module directly creates or updates `UTexture2D` assets.
- Fill each texture by computing the distance between the center coordinate and each pixel center.
- `T_InteractionFilledCircle`
  - size `64x64`
  - pixels where `distance <= 16` use `RGBA = 255,255,255,255`
  - all other pixels use `RGBA = 255,255,255,0`
- `T_InteractionOuterRing`
  - size `64x64`
  - pixels where `distance >= 20 && <= 24` use `RGBA = 255,255,255,255`
  - all other pixels use `RGBA = 255,255,255,0`
- The widget must be able to display the desired shapes using only default image brushes without any material.

### 5. Test Blueprint generation method
- `BP_Interactable_Apple`
  - parent is a test base actor or an `AActor`-based BP
  - `StaticMeshComponent` uses `/Game/Vox/Meshes/Food/SM_Vox_Apple`
  - attach `UCodexInteractionComponent`
  - interaction type is `Eat`
  - display text is `Eat`
- `BP_Interactable_Strawberry`
  - `StaticMeshComponent` uses `/Game/Vox/Meshes/Food/SM_Vox_Strawberry`
  - all other rules match the apple
- Both Blueprints must have a path that implements `UCodexInteractionTarget`.

## Agent Operating Model For Reducing Lead Time

### Base policy
- Split parallelizable work aggressively across agents.
- However, parallel work that modifies the same file and the same Blueprint at the same time is prohibited.
- Shared headers, `Build.cs`, and changes to shared data-asset structure are owned by the main worker.

### Recommended role split
- Agent A: runtime interaction types and component
  - `UCodexInteractionComponent`
  - `ECodexInteractionType`
  - `ECodexInteractionWidgetState`
- Agent B: global subsystem and input routing
  - `UCodexInteractionSubsystem`
  - player-controller input binding
  - closest-target selection logic
- Agent C: UMG base widget and state interpolation
  - `UCodexInteractionIndicatorWidget`
  - fade/scale rules per widget state
- Agent D: editor module and commandlet
  - generation code for `T_InteractionFilledCircle` and `T_InteractionOuterRing`
  - generation code for `WBP_InteractionIndicator`
  - generation code for `BP_Interactable_Apple` and `BP_Interactable_Strawberry`
  - `UCodexInteractionAssetBuildCommandlet`
- Agent E: test-asset verification
  - verify state transitions in PIE
  - verify single-prompt display
  - verify request/end debug strings
- Agent F: evaluation-only
  - separate from implementation agents
  - review structural violations, omissions, and weak parallelization

### Conditions before parallel work starts
- The main worker first locks the interfaces below:
  - `ECodexInteractionType`
  - `ECodexInteractionWidgetState`
  - `FCodexInteractionRequest`
  - function signatures of `UCodexInteractionTarget`
- After those interfaces are fixed, Agents A/B/C/D can be deployed in parallel.
- `Build.cs`, shared asset-path constants, and shared header renames are applied sequentially by the main worker.

## Prohibited
- letting the interaction component directly execute `Eat` logic or actor destruction
- letting each component independently decide whether it is the closest target
- showing prompt text boxes on multiple targets at the same time
- depending on manual import of external PNGs for the circular textures
- introducing a separate input system in parallel when an existing `Enhanced Input` structure already exists
- fabricating a new input system by guesswork and continuing when no existing `Enhanced Input` implementation exists
- ignoring the existing `IMC_` and adding a separate interaction-only `IMC_`
- creating the `Widget Blueprint` only manually in the editor and skipping the code-generation path
- generating assets while assuming an open editor UI session
- requiring handcrafted BP animation assets for every state transition
