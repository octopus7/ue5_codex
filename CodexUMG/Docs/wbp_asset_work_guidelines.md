# WBP 클래스 및 애셋 작업 지침

## 문서 목적
- 이 문서는 향후 `Widget Blueprint` 생성, 수정, 참조 방식에 대한 고정 규칙만 정의한다.
- 이 문서는 실제 WBP 생성, 애셋 수정, 커맨드렛 실행 자체를 수행하지 않는다.
- 기존 WBP 관련 계획 문서에 모호한 표현이 있더라도, WBP 애셋 참조 방식과 생성 실행 규칙에 대해서는 이 문서를 우선 기준으로 삼는다.

## 고정 원칙

### 1. 위젯 트리는 코드로 런타임 생성하지 않는다
- `WidgetTree`의 루트와 하위 위젯 구조는 실제 `WBP_*` 애셋 안에 저장되어 있어야 한다.
- 다른 시스템이 사용하는 최종 UI 구조는 반드시 UMG 디자이너에서 확인 가능한 실제 `Widget Blueprint` 형태여야 한다.
- C++ 베이스 클래스는 상태 반영, 데이터 바인딩, 애니메이션, 표시 제어를 담당할 수 있다.
- 하지만 C++가 `NativeConstruct`, `RebuildWidget`, 기타 런타임 경로에서 위젯 계층 전체를 새로 조립하는 방식은 금지한다.

### 2. 다른 곳에서 사용할 때도 클래스가 아니라 WBP 애셋을 기준으로 참조한다
- 다른 시스템, 설정값, 경로 상수, 문서, 자동화 코드에서 WBP를 가리킬 때는 `_C` 클래스 경로가 아니라 실제 `Widget Blueprint` 애셋 경로를 기준값으로 둔다.
- 기준 예시는 다음과 같다.
  - 허용: `/Game/UI/Interaction/WBP_InteractionIndicator.WBP_InteractionIndicator`
  - 금지: `/Game/UI/Interaction/WBP_InteractionIndicator.WBP_InteractionIndicator_C`
- 어떤 UE API가 마지막 단계에서 `UClass`를 요구하더라도, 저장과 참조의 원천 값은 WBP 애셋이어야 한다.
- 즉 클래스 해석이 필요하면 최종 사용 지점에서만 한 번 파생 클래스를 얻고, 시스템 전반에 `_C` 경로를 기준값으로 퍼뜨리지 않는다.

### 3. WBP 생성 및 갱신은 에디터 모듈 코드에서만 수행한다
- 실제 WBP 애셋 생성과 갱신 로직은 `CodexUMGBootstrapEditor` 같은 에디터 모듈 코드에 있어야 한다.
- 수동 에디터 클릭을 기준 작업 경로로 삼지 않는다.
- 런타임 모듈이 WBP 구조를 생성하는 방식도 허용하지 않는다.

### 4. 실행 진입점은 커맨드렛으로 고정한다
- WBP 애셋 생성/갱신 작업의 실행은 `Commandlet`로만 수행한다.
- 즉 구현은 에디터 모듈에 두고, 실제 실행 진입은 커맨드렛이 담당한다.
- 반복 가능한 자동화 경로는 항상 `UnrealEditor-Cmd.exe ... -run=<Commandlet>` 형태를 기준으로 설계한다.

### 5. 에디터 실행 충돌이 나면 우회하지 말고 즉시 중단한다
- 커맨드렛 실행 중 현재 프로젝트의 `UnrealEditor.exe` 실행으로 인해 충돌, 잠금, 저장 실패, 세션 중복 문제가 발생하면 작업을 즉시 중단한다.
- 이 경우 다른 우회 방법을 시도하지 않는다.
- 금지 예시는 다음과 같다.
  - 이미 열린 에디터 세션에 붙어서 계속 진행
  - 수동 클릭으로 같은 작업을 대신 수행
  - 다른 임시 스크립트나 별도 실행 경로로 재시도
  - 일부 애셋만 먼저 만들고 나머지를 강행
- 이 상황에서는 실패 사실과 원인을 사용자에게 그대로 알리고, 사용자가 에디터를 정리한 뒤 다시 실행하도록 안내한다.

## 권장 작업 흐름
1. WBP 생성/갱신 로직은 `CodexUMGBootstrapEditor` 내부 함수로 구현한다.
2. 해당 로직을 호출하는 전용 `Commandlet` 진입점을 만든다.
3. 커맨드렛이 실행되면 대상 WBP 애셋을 생성하거나 갱신한다.
4. 결과물은 실제 `Content` 아래의 `WBP_*` 애셋으로 남아야 한다.
5. 다른 시스템은 해당 결과물을 클래스 경로가 아니라 WBP 애셋 기준으로 참조한다.
6. 실행 중 에디터 세션 충돌이 확인되면 즉시 실패 처리하고 사용자에게 알린다.

## 금지 사항 요약
- 위젯 트리를 코드로만 만들고 WBP 애셋에는 빈 구조만 남기는 방식
- 다른 시스템에서 `_C` 클래스 경로를 WBP의 기준 참조로 사용하는 방식
- 에디터 모듈이 아닌 런타임 코드에서 WBP 애셋 생성/수정까지 맡기는 방식
- 커맨드렛이 아닌 수동 에디터 작업을 정식 생성 경로로 삼는 방식
- 에디터 실행 충돌 발생 후 우회 실행을 계속 시도하는 방식

## 완료 기준
- 대상 UI는 실제 `WBP_*` 애셋으로 프로젝트 `Content` 안에 존재한다.
- 위젯 계층은 해당 WBP 애셋의 `WidgetTree` 안에서 확인 가능하다.
- 다른 사용 지점은 WBP 클래스가 아니라 WBP 애셋을 기준으로 연결된다.
- 생성/갱신 로직은 에디터 모듈 코드에 있고 실행은 커맨드렛으로만 이루어진다.
- 에디터 실행 충돌 시 작업은 즉시 중단되고 사용자 통지가 우선된다.
