# Docs README

This guide summarizes the role of each planning document in `Docs` and the recommended reading order.

## Recommended Reading Order

1. [multi_plan_batch_execution_plan_EN.md](./multi_plan_batch_execution_plan_EN.md)
   This is the top-level execution plan for running multiple implementation plans together. It explains which plans are prerequisites, what can run in parallel, and how to avoid conflicts in shared files and Blueprints.

2. [topdown_fixed_camera_wasd_plan_EN.md](./topdown_fixed_camera_wasd_plan_EN.md)
   This is the foundation plan for building a fixed top-down camera and `WASD` movement structure. It covers `Enhanced Input`, `DA_`-based input references, access through `GameInstance`, commandlet-based asset generation, and the rule of opening the editor only after the project reaches a testable state.

3. [player_projectile_firing_plan_EN.md](./player_projectile_firing_plan_EN.md)
   This is the high-level implementation plan for letting the player fire projectiles. It defines projectile Blueprints, firing input, `DA_`-based reference wiring, `GameInstance` integration, and the rule that evaluation agents must stay separate from implementation agents.

4. [interaction_umg_component_plan_EN.md](./interaction_umg_component_plan_EN.md)
   This is the component implementation plan for showing an interaction-ready UMG widget when the player approaches a target. It includes real widget asset generation, global subsystem routing, single-prompt selection, test apple/strawberry Blueprints, and guidance for parallel agent execution.

5. [vox_mesh_asset_pipeline_plan_EN.md](./vox_mesh_asset_pipeline_plan_EN.md)
   This is the pipeline plan for converting `.vox` source files into UE5 `StaticMesh` assets. It organizes the shared VOX material, vertex-color usage, manifest structure, VOX-specific commandlet, and output paths.

6. [interaction_message_popup_plan_EN.md](./interaction_message_popup_plan_EN.md)
   This is the implementation plan for a central message popup opened from a wooden-sign interaction. It defines the real popup `Widget Blueprint` asset, `Space`-to-close input, `OK` / `YesNo` button layouts, and the result-routing structure back through the interaction subsystem.

## Per-File Summary

### [multi_plan_batch_execution_plan_EN.md](./multi_plan_batch_execution_plan_EN.md)

- This is the top-level batch execution plan that ties together `topdown_fixed_camera_wasd_plan`, `interaction_umg_component_plan`, `interaction_message_popup_plan`, `player_projectile_firing_plan`, and `vox_mesh_asset_pipeline_plan`.
- It defines a structure where `topdown_fixed_camera_wasd_plan` acts as the leading gate, while `vox_mesh_asset_pipeline_plan` runs as a separate parallel track.
- It treats `interaction_message_popup_plan` as a later integration track that starts only after both `interaction_umg_component_plan` and `vox_mesh_asset_pipeline_plan` are complete.
- It explains responsibility boundaries between the main worker and parallel worker agents, how to avoid shared-file conflicts, and the final verification flow.

### [topdown_fixed_camera_wasd_plan_EN.md](./topdown_fixed_camera_wasd_plan_EN.md)

- This is the baseline document for building a fixed north-aligned top-down camera with `WASD` movement.
- It proposes a structure where input assets such as `IA_` and `IMC_` are created as real assets, references are collected in `DA_`, and accessed through `BP_GameInstance`.
- It includes the rule of finishing asset generation and default setup through a commandlet instead of manual editor menus, then opening the editor once the project becomes testable.

### [player_projectile_firing_plan_EN.md](./player_projectile_firing_plan_EN.md)

- This document designs the player firing feature in a way that is not tightly coupled to the current implementation and remains extensible.
- Projectile Blueprints and related settings are stored in `DA_`-family assets, and the player is explicitly prevented from holding a projectile class directly.
- It defines the rule of reusing the existing input pipeline first, extending input only when necessary, and keeping implementation agents separate from evaluation agents.

### [interaction_umg_component_plan_EN.md](./interaction_umg_component_plan_EN.md)

- This is the plan document for a component that shows interaction availability as a world-space UMG marker based on player proximity.
- It defines a structure where the real `Widget Blueprint` asset is generated by editor-module code and a `Commandlet`, while a global subsystem selects exactly one nearest target as the `Interactable` state.
- It also includes test apple/strawberry Blueprints, the `Eat` interaction, request-focused screen debug rules, and guidance for running parallel agents.

### [vox_mesh_asset_pipeline_plan_EN.md](./vox_mesh_asset_pipeline_plan_EN.md)

- This is the pipeline document for producing low-resolution test VOX meshes and automatically generating UE5 `StaticMesh` assets from them.
- It fixes the constraints of `32 x 32 x 32` resolution, vertex-color representation, one shared material, and commandlet-only generation without editor UI.
- It explains source art locations, target mesh categories, UE asset output paths, the manifest, and the automation scope.
- It also includes a high-saturation diagnostic mesh such as `SM_Vox_RainbowDiagnostic` to distinguish low-saturation source palettes from color-space bugs.
- In the current process, it also covers the operational step of increasing overall palette saturation and generating angled PNG previews with `GenerateVoxPreviewPngs.py`.

### [interaction_message_popup_plan_EN.md](./interaction_message_popup_plan_EN.md)

- This is the plan document for a center-screen message popup opened through a wooden-sign interaction.
- It follows the rule that the real `WBP_InteractionMessagePopup` asset must be generated by editor-module code and a `Commandlet`, and that the widget tree must exist inside the WBP asset rather than being built at runtime.
- It defines `OK` / `YesNo` button layouts, close-button and `Space` closing, and a structure where popup results are first delivered to `UCodexInteractionSubsystem`.
- Because it reuses the existing interaction component and the VOX wooden-sign mesh, it is best read after `interaction_umg_component_plan_EN.md` and `vox_mesh_asset_pipeline_plan_EN.md`.
