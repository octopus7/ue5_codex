# 하네스 실행 계획

## 운영 원칙

- 메인 에이전트는 항상 현재 목표, 범위, 종료 조건을 먼저 고정한다.
- generator는 구현과 초안 생산에 집중한다.
- evaluator는 리뷰, 리스크 점검, 검증 관점 평가에 집중한다.
- evaluator가 이전 배치를 검토하는 동안 generator는 다음 배치를 준비할 수 있다.
- 동일 파일을 동시에 수정하는 병렬화는 금지한다.

## 단계별 계획

### 1. 하네스 기반 정리

- `Docs/하네스seed`와 `Docs/하네스`의 역할을 고정한다.
- 복원 절차와 문서 갱신 규칙을 확정한다.

### 2. 프로젝트 구조 준비

- editor 전용 모듈 추가 계획을 세운다.
- voxel source, generated asset, shared material 경로를 확정한다.
- 입력 자산, 데이터 자산, `GameInstance` 접근 경로를 확정한다.
- 미니맵 위젯 경로, 타일 데이터 경로, 캐시 소유 주체를 확정한다.

### 3. 입력 및 데이터 자산 구조

- `Enhanced Input` 사용을 전제로 `IA_`, `IMC_`, `DA_` 책임을 나눈다.
- `DA_`가 `IA_`와 `IMC_` 참조를 보유하도록 설계한다.
- `GameInstance`가 `DA_`를 소유하고 플레이어 관련 클래스에 제공하는 접근 경로를 정한다.

### 4. voxel 변환 파이프라인

- `32^3` voxel 입력 규격을 고정한다.
- voxel 파서, 메시 베이커, 자산 생성기를 editor 코드로 구현한다.
- 색상은 머터리얼 슬롯이 아니라 `vertex color`로 bake한다.

### 5. 공용 렌더링 경로

- 공용 머터리얼이 `vertex color`를 그대로 렌더링하도록 설계한다.
- 생성 자산이 동일 머터리얼 경로를 재사용하도록 강제한다.

### 6. 게임플레이 연결

- 플레이어, 적, 사물은 generated asset을 참조한다.
- 벽은 voxel 파이프라인을 거치지 않고 cube 자산 또는 단순 메시 조합을 사용한다.
- 플레이어는 탑다운 `WASD` 이동을 사용한다.
- 조준은 마우스 커서 기준으로 처리한다.
- 공격은 투사체와 근접 모드를 전환 가능한 구조로 연결한다.

### 7. 미니맵 UI 및 캐시 구조

- `WBP_` 미니맵 위젯의 배치 방식과 화면 구석 위치 제어 방법을 정한다.
- 타일 단위 데이터 구조와 dirty tile 판정 기준을 정한다.
- 캐시 저장 위치와 수명 주기를 정한다.
- 미니맵 UI가 캐시된 타일만 소비하도록 갱신 경로를 정한다.

### 8. 검증

- 생성 결과가 한 개의 공용 렌더링 경로를 쓰는지 확인한다.
- 색상별 머터리얼이 생성되지 않았는지 확인한다.
- 복원 절차로 문서 세트를 다시 세팅할 수 있는지 확인한다.
- `DA_`를 통해 `IA_`, `IMC_`를 접근하는지 확인한다.
- `GameInstance`를 통해 플레이어 관련 클래스가 입력 설정에 접근하는지 확인한다.
- 공격 전환과 공격 실행 호출 경로가 일관된지 확인한다.
- 미니맵이 `WBP_` 기반으로 원하는 화면 위치에 배치되는지 확인한다.
- 전체 redraw 대신 dirty tile만 갱신하는지 확인한다.
- 캐시된 타일이 재사용되고 불필요한 재생성이 없는지 확인한다.

## 병렬화 기준

- 병렬로 나눌 때는 쓰기 범위를 명시한다.
- editor 툴링, 입력 구조, runtime 연결, 미니맵 구조, 문서 검증은 보통 분리 가능하다.
- evaluator는 generator의 결과물, 계획, 검증 로그를 받아 비동기 검토한다.
- 메인 에이전트는 병렬 작업 결과를 수집해 하나의 의사결정으로 합친다.
