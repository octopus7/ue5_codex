# CodexMazeWalker 하네스 브리프

## 목적

이 문서는 `CodexMazeWalker`에서 장기 수행 작업을 반복 가능하게 지시하기 위한 작업 기준 문서다. 새 세션에서도 `Docs/하네스seed`만으로 `Docs/하네스`를 복원하고 다시 수행할 수 있어야 한다.

## 최상위 규칙

- 실제 작업 문서는 `Docs/하네스`에서 유지한다.
- `Docs/하네스seed`는 복원용 기준본으로 작업 중 불변이다.
- 토큰 소모보다 리드타임 단축을 우선한다.
- 병렬 수행 가능한 작업은 가능한 한 분리한다.
- 병렬 작업은 반드시 쓰기 범위가 겹치지 않도록 쪼갠다.
- generator와 evaluator가 필요하면 분리된 컨텍스트로 운영한다.
- generator와 evaluator는 직접 통신하지 않는다.
- 메인 에이전트가 결과, 피드백, 결정 사항을 중계한다.

## 핵심 기술 요구사항

- 플레이어 캐릭터, 적 캐릭터, 맵 내 사물은 `32 x 32 x 32` voxel 기준으로 제작한다.
- 벽은 voxel 파이프라인 대상이 아니다. 단순 cube 메시 조합으로 구성한다.
- voxel 원본은 에디터 코드로 게임 내 에셋으로 변환할 수 있어야 한다.
- voxel 메시 변환 시 색 정보는 색상별 머터리얼로 쪼개지 않는다.
- 메시 변환 시 색상은 `vertex color`에 기록한다.
- 렌더링은 `vertex color`를 표시하는 공용 머터리얼 하나 또는 소수의 공용 인스턴스 계열로 처리한다.
- 색상별 머터리얼 자동 생성은 금지한다.
- 입력 시스템은 `Enhanced Input`을 사용한다.
- 입력 액션과 매핑 컨텍스트는 애셋 기반으로 유지하며, `IA_`와 `IMC_` prefix를 사용하는 형태를 전제로 한다.
- 입력 관련 `IA_`와 `IMC_` 참조는 `DA_` prefix의 데이터 애셋에 묶을 수 있어야 한다.
- 해당 `DA_`는 `GameInstance`에 연결되고, 플레이어 관련 클래스가 일관된 경로로 접근할 수 있어야 한다.
- 플레이어 조작은 탑다운 기준 `WASD` 이동, 마우스 커서 조준, 원거리 투사체 발사, 근접 공격 전환을 지원해야 한다.
- 원거리/근접 전환 상태와 실제 공격 실행 경로는 플레이어 관련 클래스에서 일관된 인터페이스로 다뤄야 한다.
- 미니맵은 `WBP_` 계열 widget blueprint로 구성할 수 있어야 한다.
- 미니맵은 화면 구석의 원하는 위치에 배치할 수 있어야 하며, 앵커와 오프셋을 데이터 또는 설정으로 제어할 수 있어야 한다.
- 미니맵은 전체를 매 프레임 새로 그리지 않고 타일 단위로 필요할 때만 갱신해야 한다.
- 미니맵 갱신 결과는 캐싱되어야 하며, 변경이 없는 타일은 재사용해야 한다.
- 미니맵 갱신 구조는 성능을 우선하며, 빈번한 전체 재생성 또는 전체 redraw를 피해야 한다.

## 권장 구조

- `Source/CodexMazeWalker/`: 런타임 게임플레이 코드
- `Source/CodexMazeWalkerEditor/`: voxel import, bake, asset generation을 담당하는 editor 전용 모듈
- `Source/CodexMazeWalker/Systems/Input/`: `Enhanced Input`, 입력 바인딩, 전투 모드 전환 처리
- `Source/CodexMazeWalker/Systems/Combat/`: 투사체/근접 공격 상태와 실행 처리
- `Source/CodexMazeWalker/Systems/Game/`: `GameInstance`와 전역 데이터 접근 규약
- `Source/CodexMazeWalker/Systems/UI/`: 미니맵 데이터 수집, 타일 캐시, widget 구동
- `SourceArt/Vox/`: 원본 `.vox` 또는 동등한 voxel 소스 보관 경로
- `Content/Voxel/Materials/`: 공용 vertex color 머터리얼
- `Content/Voxel/Generated/`: 에디터 변환 결과로 생성된 메시/에셋
- `Content/Voxel/Data/`: 런타임 참조용 데이터 에셋 또는 정의 자산
- `Content/Environment/Walls/`: cube 기반 벽 자산 또는 벽용 블루프린트
- `Content/Input/`: `IA_` 및 `IMC_` 계열 입력 애셋
- `Content/Data/`: `DA_` 계열 데이터 애셋
- `Content/UI/`: `WBP_` 계열 UI 애셋과 미니맵 위젯

## 입력 및 데이터 자산 규약

- `IA_`는 개별 입력 액션 애셋이다.
- `IMC_`는 플레이어 입력 조합을 묶는 mapping context 애셋이다.
- `DA_`는 플레이어 입력 참조와 관련 설정을 담는 데이터 애셋이다.
- `DA_`는 `GameInstance`에 한 번 연결하고, player controller, pawn, character 등 플레이어 관련 클래스는 `GameInstance`를 통해 접근한다.
- 플레이어 관련 클래스는 `IA_`나 `IMC_`를 직접 하드코딩하지 않고 `DA_` 경유 접근을 우선한다.
- 입력 계층과 전투 계층이 분리되도록, 입력은 의도를 생성하고 실제 공격 실행은 별도 전투 로직이 담당한다.

## 플레이어 전투 루프 요구사항

- 기본 이동은 탑다운 시점의 `WASD` 평면 이동이다.
- 마우스 커서 월드 위치 또는 적절한 히트 결과를 기준으로 조준한다.
- 플레이어는 투사체 공격 모드와 근접 공격 모드를 전환할 수 있어야 한다.
- 현재 활성 공격 모드에 따라 공격 입력이 다른 실행 경로를 타더라도 호출 지점은 일관되어야 한다.
- 전투 모드 전환 상태는 UI, 애니메이션, 무기 처리 확장에 견딜 수 있는 형태로 유지한다.

## 미니맵 요구사항

- 미니맵은 `WBP_` 기반 위젯으로 제공한다.
- 미니맵 위젯은 화면의 원하는 구석에 배치 가능해야 하며, 추후 위치 프리셋 또는 사용자 설정 확장에 견딜 수 있어야 한다.
- 미니맵 데이터는 타일 단위로 관리한다.
- 플레이어 이동, 타일 발견, 시야 변화, 맵 상태 변경 등 실제 필요 이벤트가 있을 때만 관련 타일을 갱신한다.
- 타일 캐시는 재사용 가능해야 하며, 변동 없는 타일은 다시 생성하지 않는다.
- UI 갱신은 캐시된 타일 데이터를 소비하는 구조여야 하며, 매 프레임 전체 맵을 다시 그리는 방식은 피한다.

## 현재 코드베이스 기준 참고

- 현재 프로젝트에는 `Source/CodexMazeWalker/` 런타임 모듈만 존재한다.
- voxel 에디터 변환 파이프라인을 위해 editor 모듈 추가를 전제로 계획을 세운다.

## 최소 산출물

- voxel 소스 로더 또는 importer
- voxel 데이터를 mesh로 bake하는 editor 코드
- `vertex color` 기반 공용 머터리얼
- 생성 자산 경로와 재생성 기준을 담는 규약
- 플레이어, 적, 사물용 voxel 자산 생성 경로
- 벽을 cube로 유지하는 별도 환경 제작 규약
- `Enhanced Input` 기반 입력 자산 참조 구조
- `DA_` to `GameInstance` to player class 접근 경로
- 탑다운 이동, 커서 조준, 공격 모드 전환, 공격 실행 구조
- `WBP_` 기반 미니맵 위젯 구조
- 타일 단위 갱신과 캐시 구조를 가진 미니맵 데이터 경로

## 완료 조건

- `Docs/하네스seed`에서 `Docs/하네스`를 재생성할 수 있다.
- generator/evaluator 분리 운영 규칙이 문서화되어 있다.
- voxel 자산 요구사항과 벽 예외 규칙이 명확하다.
- 공용 vertex color 머터리얼 원칙이 흔들리지 않도록 명시되어 있다.
- 입력 자산, 데이터 자산, `GameInstance` 접근 구조가 문서에 명확하다.
- 탑다운 플레이 루프와 공격 전환 구조가 문서에 명확하다.
- 미니맵의 WBP 배치 방식과 타일 캐시 기반 갱신 구조가 문서에 명확하다.
