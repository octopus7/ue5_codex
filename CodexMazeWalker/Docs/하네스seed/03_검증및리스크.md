# 검증 및 리스크

## evaluator 기본 체크리스트

- generator가 수정한 파일과 의도가 일치하는가
- editor 전용 로직이 runtime 경로에 새지 않았는가
- voxel 자산 규격이 `32 x 32 x 32` 제약을 유지하는가
- 벽이 voxel 파이프라인으로 잘못 들어가지 않았는가
- 색상별 머터리얼이 새로 생기지 않았는가
- 생성 메시가 `vertex color`를 사용하도록 설계되었는가
- 공용 머터리얼 경로가 일관되게 재사용되는가
- 복원 절차가 여전히 유효한가
- `Enhanced Input` 경로가 유지되는가
- `IA_`, `IMC_`, `DA_` 참조 관계가 깨지지 않았는가
- `DA_`가 `GameInstance`에 연결되고 플레이어 관련 클래스가 해당 경로를 통해 접근하는가
- 탑다운 이동, 커서 조준, 투사체/근접 전환 요구사항이 누락되지 않았는가
- 미니맵이 `WBP_` 기반으로 구성되는가
- 미니맵 위치가 하드코딩되지 않고 설정 가능하게 남아 있는가
- dirty tile 갱신만 수행하고 전체 redraw를 남발하지 않는가
- 타일 캐시가 실제로 재사용되고 있는가

## 주요 리스크

- `.vox` 포맷 해석 범위를 과도하게 넓히면 초기 구현 리드타임이 늘어날 수 있다.
- 생성 자산 경로와 재생성 규칙이 불안정하면 source와 generated 결과가 쉽게 어긋난다.
- `vertex color` 기반 머터리얼에서 감마, sRGB, 조명 모델 선택이 엇갈리면 원본 색 재현이 흔들릴 수 있다.
- 벽과 voxel 사물의 제작 파이프라인이 섞이면 추후 유지보수가 어려워진다.
- 병렬 작업에서 쓰기 범위가 겹치면 merge 비용이 리드타임 이득을 상쇄한다.
- 입력 애셋을 클래스 내부에 직접 하드코딩하면 데이터 주도 구성이 무너질 수 있다.
- `GameInstance` 접근 경로가 약하면 player controller, pawn, character마다 참조 방식이 갈라질 수 있다.
- 공격 모드 전환 상태와 실제 공격 실행이 결합되면 추후 무기 확장이 어려워질 수 있다.
- 미니맵 타일 캐시 소유 주체가 불명확하면 UI와 월드 데이터가 서로 강결합될 수 있다.
- dirty tile 판정이 약하면 필요 이상으로 많은 갱신이 발생해 성능 이점이 사라질 수 있다.
- 화면 위치를 위젯 내부 값으로만 고정하면 추후 사용자 지정 위치 요구를 수용하기 어렵다.
