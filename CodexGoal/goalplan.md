# Codex Goal Plan: 호랑이 떡집

작성일: 2026-05-03
프로젝트: `CodexGoal`
엔진: Unreal Engine 5.7

## 목표

`호랑이 떡집`은 고정된 3D 화면에서 진행되는 클릭커 게임이다. 플레이어는 떡집 안의 작업 지점을 클릭해 떡을 생산하고, 생산량과 자동화 능력을 업그레이드하며, 호랑이 캐릭터와 떡집 공간을 점진적으로 확장한다.

이 문서는 Codex Goal 기능 테스트를 위한 지속 관리 문서다. 작업 목표, 설계 결정, 완료 조건, 진행 기록을 이 파일에 계속 갱신한다.

## 핵심 플레이

- 플레이어는 화면의 떡 만들기 지점을 클릭한다.
- 클릭 1회는 기본 떡 생산량을 증가시킨다.
- 떡은 재화로 사용된다.
- 재화로 클릭 생산량, 자동 생산량, 떡집 설비를 업그레이드한다.
- 일정 단계 이후에는 새 작업대, 장식, 캐릭터 동작, 자동 일꾼 같은 확장 요소를 추가할 수 있다.

## 첫 번째 완성 목표

첫 번째 목표는 "작동하는 3D 클릭커 프로토타입"이다.

완료 조건:

- 기본 맵에서 고정 카메라로 떡집 공간이 보인다.
- 클릭 가능한 떡 만들기 대상이 있다.
- 클릭하면 떡 수량이 증가한다.
- 현재 떡 수량이 UI에 표시된다.
- 최소 1개 이상의 업그레이드 버튼이 있고, 구매 시 클릭당 생산량이 증가한다.
- 게임 시작 후 입력, UI, 생산량 변경이 에디터 플레이에서 확인된다.

## 화면과 카메라

초기 버전은 고정 뷰 3D 게임으로 만든다.

카메라 기본 방향:

- 떡집 작업대와 캐릭터가 한 화면에 들어오는 사선 상단 시점
- 플레이어가 월드 오브젝트를 클릭하기 쉬운 각도
- 카메라 회전은 초기 버전에서 제공하지 않음

확장성 설계:

- 카메라는 별도 `CameraRig` 액터로 관리한다.
- 초기에는 고정 위치와 고정 회전만 사용한다.
- 이후 맵이 넓어질 때를 대비해 패닝 입력, 패닝 경계, 포커스 대상 이동을 추가할 수 있게 구조를 나눈다.
- 클릭 판정과 카메라 이동 로직은 분리한다.

예상 클래스:

- `ATigerShopCameraRig`
- `UTigerShopCameraComponent` 또는 일반 `UCameraComponent`
- `ATigerShopPlayerController`

## 캐릭터 임시 구현

질문: 임의로 뼈대만 넣어서 임시 스켈레탈 메시로 구성 가능한가?

답: 가능은 하지만 "뼈대만 있는 스켈레탈 메시"는 화면에 보이지 않는다. Unreal의 `USkeletalMesh`는 렌더링할 버텍스와 스킨 웨이트가 필요하다. 순수한 뼈대만으로는 애니메이션 구조는 표현할 수 있어도 캐릭터 시각 표현은 만들 수 없다.

프로토타입 권장 방식:

- 1단계에서는 진짜 `USkeletalMesh` 대신 임시 본 구조 액터를 사용한다.
- `SceneComponent`를 뼈대처럼 계층 구성한다.
- 각 본 위치에 캡슐, 구, 박스 같은 단순 메시를 붙여 임시 호랑이 캐릭터를 보이게 한다.
- 이후 실제 호랑이 스켈레탈 메시가 생기면 같은 역할의 캐릭터 액터 안에서 교체한다.

대안:

- 정말 `USkeletalMeshComponent` 기반 테스트가 필요하면 최소 지오메트리와 스킨 웨이트가 포함된 임시 FBX 또는 엔진 내 생성 에셋이 필요하다.
- 이 경우 뼈대만이 아니라 단순 몸통 메시라도 포함해야 한다.

초기 결정:

- MVP에서는 `ATemporaryTigerCharacter`를 만든다.
- 내부는 본 이름에 해당하는 `USceneComponent` 계층과 간단한 primitive mesh 또는 static mesh component로 구성한다.
- 실제 스켈레탈 메시 교체를 고려해 입력/상태/애니메이션 트리거 인터페이스는 캐릭터 액터 단위로 숨긴다.

## 주요 시스템

### 클릭 생산

- 월드의 떡 만들기 지점을 클릭하면 `RiceCakeCount`가 증가한다.
- 클릭 가능한 대상은 Actor 또는 Component로 분리한다.
- 클릭 생산량은 업그레이드로 증가한다.

예상 클래스:

- `ARiceCakeWorkstation`
- `UTigerClickProductionComponent`
- `UTigerCurrencyComponent`

### 업그레이드

- 업그레이드는 비용과 효과를 가진다.
- 첫 버전은 코드 또는 간단한 데이터 구조로 시작한다.
- 이후 DataAsset 기반으로 확장 가능하게 한다.

첫 업그레이드 예시:

- 이름: 떡메 강화
- 비용: 떡 10개
- 효과: 클릭당 생산량 +1

### UI

- 현재 떡 수량 표시
- 클릭당 생산량 표시
- 업그레이드 구매 버튼 표시

초기 UI는 기능 확인을 우선한다. 시각 디자인은 떡집 분위기가 드러나도록 하되, 첫 단계에서는 과한 장식보다 읽기 쉬운 HUD를 우선한다.

### 저장

초기 MVP에는 저장을 필수로 넣지 않는다. 클릭커 게임 특성상 두 번째 단계에서 `USaveGame` 기반 저장을 추가한다.

저장 후보 데이터:

- 떡 수량
- 클릭당 생산량
- 자동 생산량
- 구매한 업그레이드 단계

## 개발 단계

### 1단계: 기본 3D 클릭커

상태: 구현 완료. 빌드, 런타임 로드, 자동화 테스트 검증 완료. 실제 UI 클릭 감각은 에디터 플레이에서 수동 확인 권장.

목표:

- 기본 레벨 구성
- 고정 카메라 구성
- 클릭 가능한 떡 만들기 대상 구현
- 떡 수량 UI 구현
- 클릭당 생산량 업그레이드 1종 구현

완료 조건:

- 플레이 중 클릭으로 떡 수량이 증가한다.
- UI가 즉시 갱신된다.
- 업그레이드 구매 후 클릭당 생산량이 증가한다.

### 2단계: 임시 호랑이 캐릭터

상태: 구현 완료. 임시 본 계층 액터와 생산 반응 동작 추가.

목표:

- 임시 본 계층 캐릭터 액터 구현
- 클릭 또는 생산 이벤트에 반응하는 간단한 동작 추가
- 이후 실제 스켈레탈 메시 교체가 가능하도록 구조화

완료 조건:

- 화면에 임시 호랑이 캐릭터가 보인다.
- 생산 이벤트 발생 시 캐릭터 일부가 움직이거나 반응한다.

### 3단계: 카메라 확장 기반

상태: 구현 완료. 고정 카메라와 향후 패닝 API 추가.

목표:

- `CameraRig` 기반 구조 정리
- 초기 고정 시점 유지
- 향후 패닝을 위한 경계값과 포커스 이동 API 설계

완료 조건:

- 현재 게임은 고정 뷰로 동작한다.
- 카메라 코드는 패닝 추가를 막지 않는 구조다.

### 4단계: 저장과 반복 플레이

상태: 구현 완료. `USaveGame` 기반 자동 저장과 로드 추가.

목표:

- `USaveGame` 기반 저장 추가
- 재시작 후 떡 수량과 업그레이드 상태 복원

완료 조건:

- 게임 종료 후 다시 시작해도 주요 진행 상태가 유지된다.

## 현재 결정 사항

- 게임 이름은 `호랑이 떡집`으로 한다.
- 장르는 3D 클릭커다.
- 초기 카메라는 고정 뷰다.
- 패닝은 당장 구현하지 않고 확장 가능하게 설계만 한다.
- 캐릭터는 첫 단계에서 실제 스켈레탈 메시 없이 임시 본 계층 액터로 구현한다.
- 실제 스켈레탈 메시가 필요해지는 시점에는 최소 지오메트리와 스킨 웨이트가 포함된 임시 에셋을 사용한다.
