# CodexHarness 결정 로그

최종 수정일: 2026-04-04

## 사용 규칙

- 전역 제약, 구현 정책, 예외 처리 방식이 바뀌면 이 문서에 먼저 기록한다.
- 임시로 넣은 규칙이라도 다음 단계에 영향을 주면 기록한다.
- 각 항목은 날짜와 영향을 받는 영역을 함께 적는다.

## 결정 사항

### 2026-04-04 | 작업 방식 | 장기 작업은 계층형 하네스만으로 보지 않고 문서 세트로 운영

- 단일 하네스 문서만으로는 장기 작업 지속성이 부족하다고 판단했다.
- 상위 목표, 현재 단계, 결정 이력, 상태 스냅샷을 분리한 운영 문서 세트를 기본 방식으로 사용한다.

### 2026-04-04 | 실행 환경 | Unreal Editor GUI는 열지 않음

- 작업은 에디터가 꺼진 상태를 기본 전제로 한다.
- 에디터 API가 필요한 작업은 `UnrealEditor-Cmd`와 커맨드렛으로 처리한다.
- 수동 에디터 조작을 전제로 한 워크플로는 기본 경로로 사용하지 않는다.

### 2026-04-04 | 입력 시스템 | 레거시 입력 대신 EnhancedInput 사용

- 입력 시스템은 `EnhancedInput`으로 통일한다.
- 새 입력 작업은 레거시 축/액션 매핑 확장 대신 `Input Action`과 `Input Mapping Context` 기준으로 설계한다.

### 2026-04-04 | 구현 분리 | 핵심 구조는 C++, 조정과 표현은 Blueprint/UMG

- 핵심 게임플레이 구조는 C++에 둔다.
- Blueprint와 UMG는 튜닝, 레이아웃, 시각 구성, 에셋 연결에 사용한다.
- 이 분리는 탑다운 슈터 프로토타입 전체에 공통 적용한다.

### 2026-04-04 | 메시 제작 | 새 메시가 필요하면 VOX 소스부터 생성

- 새 메시가 필요할 경우 임의의 수작업 메시 대신 `.vox` 소스를 먼저 생성한다.
- 기본 해상도는 `64 x 64 x 64` 이내로 둔다.
- `.vox` 소스는 `SourceArt/Vox/` 아래에 보관하는 것을 기본 규칙으로 한다.
- 생성된 Unreal 애셋은 `/Game/TopDownShooter/Vox/` 아래에 저장하는 것을 기본 규칙으로 한다.

### 2026-04-04 | 메시 머터리얼 | VOX 메시용 베이스 머터리얼은 1개만 재사용

- VOX 메시용 베이스 머터리얼은 프로젝트 전역에서 하나만 유지한다.
- 기본 이름은 `M_VoxBase`로 한다.
- 베이스 컬러는 버텍스 컬러를 사용해서 칠할 수 있어야 한다.
- 메시별 전용 컬러 머터리얼을 늘리는 방식은 기본 경로로 사용하지 않는다.

### 2026-04-04 | 구현 우선순위 | 제작 파이프라인을 먼저 고정하고 게임플레이를 올림

- 장기 운영 검증 목적상, 커맨드렛과 애셋 생성 루트를 먼저 고정하는 것이 유리하다고 판단했다.
- 순수 C++ 게임플레이 구현은 그 위에 단계적으로 올린다.

### 2026-04-04 | 커맨드렛 호스팅 | 프로젝트 커맨드렛은 `Default` 로딩 에디터 모듈에서 제공

- 프로젝트 커맨드렛 클래스 탐색은 `LoadStartupModules` 이후, `PostEngineInit` 이전에 일어난다.
- 커맨드렛을 담는 프로젝트 에디터 모듈은 `PostEngineInit`이 아니라 `Default` 로딩 단계에 등록한다.
- 이 규칙에 따라 `CodexHarnessEditor` 모듈의 로딩 단계는 `Default`로 고정한다.

### 2026-04-04 | 헤드리스 부트스트랩 | `CHBuildHarnessAssets` 커맨드렛이 VOX 스모크 입력과 베이스 머터리얼을 보장

- 헤드리스 애셋 부트스트랩 명령 이름은 `CHBuildHarnessAssets`로 고정한다.
- 이 커맨드렛은 `SourceArt/Vox/Smoke/CH_SmokeCube.vox`를 생성하거나 갱신할 수 있어야 한다.
- 이 커맨드렛은 `/Game/TopDownShooter/Vox/Materials/M_VoxBase`를 생성 또는 재구성하고 저장할 수 있어야 한다.
- 실행 결과는 `Saved/Harness/M0_P1_BootstrapReport.txt`에 남겨 반복 실행 검증에 사용한다.

### 2026-04-04 | 기본 스폰 | `GlobalDefaultGameMode`와 폴백 스폰으로 `BasicMap` 진입을 보장

- 프로젝트 기본 게임 모드는 `CHTopDownGameMode`로 연결한다.
- 기본 스폰은 엔진의 일반 `PlayerStart` 경로를 우선 사용한다.
- `PlayerStart`가 없거나 기본 스폰이 실패하면 `CHTopDownGameMode`가 폴백 변환으로 기본 폰을 직접 생성한다.

### 2026-04-04 | 카메라 골격 | 플레이어 캐릭터는 탑다운 스프링암 카메라 구조를 기본으로 사용

- `CHPlayerCharacter`는 절대 회전 스프링암과 카메라를 기본 탑다운 시점 구조로 가진다.
- 현재 단계에서는 이동과 조준보다 카메라 배치를 먼저 고정한다.
- 후속 단계에서 입력과 회전 로직은 이 카메라 구조 위에 덧붙인다.

### 2026-04-04 | 이동 입력 연결 | `M1-P2`에서는 런타임 생성 `EnhancedInput` 객체로 `WASD`를 우선 연결

- `Input Action`과 `Input Mapping Context` 에셋이 아직 없더라도 진행을 멈추지 않기 위해, `CHPlayerController`가 런타임에 이동 액션과 매핑 컨텍스트를 생성한다.
- `W`, `A`, `S`, `D`는 하나의 2D 이동 액션으로 합성한다.
- 이후 단계에서 필요하면 같은 책임 분리를 유지한 채 헤드리스 애셋 생성 경로로 옮길 수 있다.

### 2026-04-04 | 초기 이동 구현 | 플레이어 이동은 직접 월드 오프셋 방식으로 먼저 고정

- `CHPlayerCharacter::Move`는 카메라 평면 기준 방향을 계산한 뒤 직접 월드 오프셋으로 이동한다.
- 이 구현은 초기 프로토타입 진행과 헤드리스 검증 안정성을 우선한 임시 경로다.
- 충돌 스윕, 가속, 감속 같은 이동 감각 보정은 이후 단계에서 다시 다룬다.

### 2026-04-04 | 작업 추적 | 단계 완료 시간은 `PHASE_TIME_LOG_KO.md`에 append 방식으로 누적

- 단계 시작 시 계획은 `CURRENT_PHASE_KO.md`에 유지하고, 단계 완료 시각과 소요 시간은 `Docs/Harness/PHASE_TIME_LOG_KO.md`에 한 줄씩 누적한다.
- 이미 끝난 단계의 시간은 대화 이력 기준 추정 백필로 남기고, 이후 단계부터는 실제 시각을 직접 기록한다.
- 사후 분석 시에는 이 시간 로그를 기준으로 병목 구간과 재작업 구간을 확인한다.

### 2026-04-04 | 조준 경로 | 마우스 조준은 폰 높이 평면 교차점으로 계산하고 헤드리스 스모크는 고정 목표를 사용

- `M2-P1`의 조준 좌표는 기본적으로 `DeprojectMousePositionToWorld` 레이와 플레이어 높이 평면의 교차점으로 계산한다.
- 헤드리스 검증에서는 실제 마우스 포인터가 없을 수 있으므로 `-CHAimSmoke` 옵션으로 고정 오프셋 목표를 사용한다.
- 다음 단계의 발사 로직은 캐릭터가 보관한 마지막 조준 월드 포인트를 재사용한다.

### 2026-04-04 | 발사 입력 | 기본 발사는 런타임 `EnhancedInput` 좌클릭 입력으로 연결

- `M3-P1`의 발사 입력은 별도 애셋 없이도 진행을 멈추지 않기 위해 런타임 생성 `Input Action`으로 추가한다.
- 기본 입력 키는 `LeftMouseButton`으로 고정한다.
- 이후 헤드리스 입력 애셋 생성 경로가 생기더라도 기본 발사 책임은 `CHPlayerController`가 유지한다.

### 2026-04-04 | 첫 발사 루프 | 조준 재사용형 히트스캔과 `FCHFireResult` 구조를 기본 경로로 사용

- 첫 발사 루프는 `CHPlayerCharacter::TryFire`에서 조준 월드 포인트를 재사용해 `ECC_Visibility` 라인트레이스를 쏘는 방식으로 고정한다.
- 발사 결과는 `FCHFireResult` 구조에 보관해 다음 단계의 데미지 처리와 적 피격 판정에 재사용한다.
- 헤드리스 검증은 `-CHFireSmoke` 옵션으로 자동 1회 발사 후 로그를 남기는 방식으로 수행한다.

### 2026-04-04 | 체력 처리 | 공용 체력은 `CHHealthComponent`와 엔진 `TakeDamage` 경로로 통일

- `M3-P2`부터 체력과 사망 처리는 `UCHHealthComponent`가 담당한다.
- 실제 데미지 입력은 `AActor::TakeDamage` 경로를 통해 들어오고, 플레이어 캐릭터는 이를 `UCHHealthComponent`로 전달한다.
- 이후 적과 투사체도 같은 데미지 진입점을 재사용한다.

### 2026-04-04 | 사망 검증 | 헤드리스 데미지 스모크는 3단계 자기 피해 시퀀스로 고정

- 체력 감소, 사망, 사망 후 추가 피해 무시를 한 번에 검증하기 위해 `-CHDamageSmoke` 옵션을 사용한다.
- 이 스모크는 `40 -> 70 -> 10` 데미지 시퀀스를 적용해 남은 체력과 사망 카운트를 로그로 남긴다.
- 사망은 한 번만 관측되어야 하며, 세 번째 피해는 적용되지 않아야 한다.

### 2026-04-04 | 적 스폰 | 기본 적 1종은 게임 모드가 플레이어 기준 오프셋 위치에 자동 스폰

- `M4-P1`의 기본 적은 `CHTopDownGameMode`가 플레이어 위치 기준 `FVector(400, 400, 0)` 오프셋에 스폰한다.
- 이 위치는 헤드리스 `CHAimSmoke` 고정 조준점과도 맞물려 이후 전투 검증에 재사용할 수 있다.
- 웨이브 단계에서는 이 스폰 경로를 확장한다.

### 2026-04-04 | 적 추적과 공격 | 적은 AI 에셋 없이 직접 추적 이동과 근접 공격을 사용

- `ACHEnemyCharacter`는 별도 AI 컨트롤러나 네비게이션 에셋 없이 직접 플레이어를 향해 이동한다.
- 공격은 근접 범위와 쿨다운만 가진 최소 루프로 시작한다.
- 헤드리스 검증은 `-CHEnemySmoke` 옵션으로 스폰, 추적, 첫 공격 로그를 남긴 뒤 종료한다.

### 2026-04-04 | 웨이브 관리 | 웨이브와 적 수 추적은 게임 모드가 직접 관리

- `M4-P2`부터 현재 웨이브 번호와 살아 있는 적 수는 `CHTopDownGameMode`가 직접 보관한다.
- 적이 파괴되면 `OnDestroyed`를 통해 살아 있는 적 수를 갱신한다.
- 적 수가 0이 되면 다음 웨이브를 시작한다.

### 2026-04-04 | 웨이브 스모크 | `CHWaveSmoke`는 1웨이브 제거 후 2웨이브 시작까지만 검증

- 헤드리스 웨이브 검증은 `-CHWaveSmoke` 옵션으로 수행한다.
- 이 스모크는 1웨이브 적을 자동 제거해 `Alive=0`을 만들고, 2웨이브 스폰 로그가 찍히면 종료한다.
- HUD 단계에서는 이 웨이브 상태 값을 그대로 재사용한다.

### 2026-04-04 | HUD 구현 | 최소 HUD는 `AHUD` 캔버스 텍스트로 먼저 고정

- 에디터 GUI 없이 즉시 확인 가능한 경로를 우선하기 위해 `M5-P1` HUD는 `ACHGameHUD`의 `DrawHUD` 텍스트 렌더링으로 구현한다.
- HUD는 플레이어 체력과 게임 모드의 웨이브, 적 수 상태를 직접 읽는다.
- 헤드리스 검증은 `-CHHudSmoke` 옵션으로 같은 데이터 경로를 로그에 남기는 방식으로 수행한다.

### 2026-04-04 | 게임 오버 상태 | 게임 오버 플래그는 게임 모드, 재시작 입력은 플레이어 컨트롤러가 담당

- 플레이어 사망 시 `CHTopDownGameMode`가 `CHGameOverState=1`로 게임 오버 상태를 확정한다.
- HUD는 이 상태를 읽어 `GAME OVER`와 재시작 안내를 그린다.
- 재시작 입력은 `R` 키와 `CHRestartSmoke` 자동 경로 모두 `CHPlayerController`를 통해 들어간다.
