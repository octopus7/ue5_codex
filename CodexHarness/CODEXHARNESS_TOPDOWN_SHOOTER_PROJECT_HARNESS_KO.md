# CodexHarness 탑다운 슈터 프로젝트 하네스

이 파일은 현재 `d:\github\ue5_codex\CodexHarness` 프로젝트를 기준으로 채운 실제 작업용 프로젝트 하네스다.

현재 프로젝트는 다음 상태다.
- UE `5.7` 프로젝트다.
- `GameDefaultMap`과 `EditorStartupMap`은 `/Game/Maps/BasicMap`으로 설정되어 있다.
- `BasicMap`은 존재한다.
- 런타임 C++ 모듈은 존재하지만 게임플레이 클래스는 아직 없다.
- `EnhancedInput` 모듈 의존성은 추가되어 있지만 실제 입력 액션, 매핑 컨텍스트, 바인딩 구현은 없다.
- 플레이어 이동, 조준, 발사, 적, HUD, 게임 흐름은 구현되지 않았다.
- Unreal Editor GUI는 실행하지 않는 전제로 작업한다.

아래 블록은 이 프로젝트에 대해 그대로 복사해서 사용할 수 있는 실제 요청문이다.

## 실제 요청문

```md
먼저 이 파일을 읽어줘:
d:\github\ue5_codex\CodexHarness\CODEXHARNESS_TOPDOWN_SHOOTER_PROJECT_HARNESS_KO.md

그다음 이 파일을 현재 UE5 프로젝트의 작업용 프로젝트 하네스로 사용해줘.

프로젝트 경로:
d:\github\ue5_codex\CodexHarness

프로젝트 목표:
현재 거의 비어 있는 CodexHarness 프로젝트를 싱글플레이 PC용 탑다운 슈터 프로토타입으로 바닥부터 구축한다.

현재 프로젝트 상태:
- 입력 시스템: `EnhancedInput` 모듈만 연결되어 있고 실제 `Input Action`, `Input Mapping Context`, 플레이어 바인딩은 없다.
- 플레이어 폰과 이동: 구현 없음.
- 카메라와 조준: 구현 없음.
- 무기와 전투 루프: 구현 없음.
- 적과 AI 루프: 구현 없음.
- HUD와 UI: 구현 없음.
- 웨이브 흐름: 구현 없음.
- 저장, 설정, 진행도: 구현 없음.
- 프로젝트 맵: `/Game/Maps/BasicMap`만 존재한다.
- 코드 구조: 기본 런타임 모듈 파일만 있고 `GameMode`, `PlayerController`, `Character`, `Enemy`, `Widget` 관련 클래스가 없다.

목표 플레이 가능 결과물:
플레이어가 `BasicMap`에서 스폰되고, `WASD`로 이동하며, 마우스 커서를 향해 몸을 회전하고, 좌클릭으로 기본 총기를 발사할 수 있어야 한다.
기본 적 1종이 일정 간격으로 스폰되어 플레이어를 추적하고 근접 공격을 해야 한다.
플레이어와 적은 체력을 가지며 피격과 사망이 동작해야 한다.
화면에는 체력, 현재 웨이브, 남은 적 수가 표시되어야 한다.
플레이어 사망 시 게임 오버 상태가 표시되고 재시작이 가능해야 한다.

게임플레이 핵심 축:
- 반응성이 좋은 이동, 조준, 발사 감각
- 최소 시스템으로도 반복 플레이가 가능한 생존형 웨이브 루프
- C++ 중심 기반 구조 위에 Blueprint와 UMG로 조정 가능한 설계

카메라와 조작 스타일:
- 카메라 모델: 고정 각도의 Follow Top-Down Camera
- 이동 모델: `WASD`
- 조준 모델: 마우스 커서 기반 월드 위치 조준
- 발사 모델: 히트스캔 기반 기본 총기 1종

플랫폼 및 엔진 제약:
- 엔진 버전: `UE 5.7`
- 대상 플랫폼: `PC`
- 네트워크 모델: `Single Player`
- 성능 목표: 보통 사양 데스크톱 기준 `60 FPS`

구현 규칙:
- C++ 담당 범위:
  - 핵심 게임플레이 클래스
  - 게임 모드와 게임 상태 흐름
  - 플레이어 캐릭터, 플레이어 컨트롤러
  - 적 액터와 추적 또는 공격 로직
  - 체력, 피격, 사망, 데미지 처리
  - 무기 발사와 히트 판정
  - 웨이브 스폰과 라운드 진행
  - UI에 전달할 상태 데이터와 이벤트
- Blueprint 담당 범위:
  - 캐릭터 메쉬, 머티리얼, 이펙트, 사운드 연결
  - 카메라 수치, 이동 속도, 발사 속도, 체력 값 같은 밸런스 튜닝
  - 위젯 레이아웃과 시각 표현
  - 입력 에셋(`Input Action`, `Input Mapping Context`) 생성과 연결
  - 맵 배치와 스폰 포인트 배치
- 폴더 및 네이밍 규칙:
  - C++ 클래스 접두사는 `CH`를 사용한다.
  - 소스 코드는 최소한 `Core`, `Input`, `Player`, `Combat`, `Enemy`, `UI`, `GameFlow` 단위로 정리한다.
  - 콘텐츠 경로는 `/Game/TopDownShooter/...` 하위로 정리한다.
  - 클래스 예시:
    - `CHTopDownGameMode`
    - `CHPlayerController`
    - `CHPlayerCharacter`
    - `CHHealthComponent`
    - `CHWeaponComponent`
    - `CHEnemyCharacter`
    - `CHWaveSpawner`
    - `CHHUDWidget` 또는 이에 대응하는 위젯 Blueprint
- 플러그인 규칙:
  - 외부 플러그인은 추가하지 않는다.
  - 기본 제공 기능과 현재 프로젝트 설정만 사용한다.
  - UI는 우선 `UMG` 중심으로 구현하고, `CommonUI` 의존은 꼭 필요할 때만 추가 검토한다.
- 에셋 생성 규칙:
  - Unreal Editor GUI는 열지 않는다.
  - 애셋 생성이 단순 파일 복사나 설정 편집만으로 안전하게 처리되지 않으면, 에디터 코드를 직접 실행하는 커맨드렛 경로를 사용한다.
  - 필요하면 에디터 전용 모듈 또는 커맨드렛 클래스를 추가하고, `UnrealEditor-Cmd.exe <uproject> -run=<CommandletName>` 방식으로 실행한다.
  - `Input Action`, `Input Mapping Context`, Blueprint, Widget Blueprint, Data Asset, 맵 수정처럼 에디터 API가 필요한 작업은 기본적으로 이 경로를 우선 검토한다.
- 호환성 제약:
  - 기존 프로젝트 루트, 모듈명 `CodexHarness`, 기본 맵 `/Game/Maps/BasicMap`은 유지한다.
  - UE 5.7 기준으로 빌드 가능해야 한다.
  - 확인 없이 멀티플레이 전용 구조를 먼저 도입하지 않는다.

프로젝트 범위:
- 포함 범위:
  - 플레이어 입력과 이동
  - 탑다운 카메라와 마우스 조준
  - 기본 총기 1종
  - 체력, 피격, 사망
  - 적 1종의 추적과 근접 공격
  - 웨이브 기반 스폰
  - 최소 HUD
  - 게임 오버와 재시작
  - 플레이 가능한 프로토타입 완성까지의 구조화된 구현
- 제외 범위:
  - 멀티플레이

  - 설정 메뉴

- 한 번에 전부 구현하려 하지 말고, 플레이 가능한 단위로 끊어서 진행해줘.
- 각 단계에서 파일을 수정하기 전에 정확한 작업 범위와 수정 예정 파일을 먼저 설명해줘.
- 각 단계가 끝나면 아래를 반드시 보고해줘:
  - 무엇이 구현되었는지
  - 어떤 파일이 바뀌었는지

  - 다음 추천 단계가 무엇인지
- 에셋이 없으면 엔진 기본 도형, 기본 머티리얼, 임시 위젯으로 대체해서 진행해줘.

- 막히는 지점이 생기면 가능한 한 코드와 설정으로 먼저 해결하고, 정말 결정이 필요한 경우에만 짧은 질문 하나를 해줘.

현재 실행 경계:
- 시작 마일스톤: `1. 기반 구성`

  - 무기, 적, HUD, 웨이브는 아직 구현하지 않는다.
- 내가 명시적으로 요청하지 않으면 그 다음 마일스톤으로 넘어가지 마.

반드시 지켜야 하는 제약:
- 입력은 레거시 입력이 아니라 `EnhancedInput`으로 구현해줘.
- 핵심 게임플레이 구조는 C++ 중심으로 작성해줘.
- 에셋 부족을 이유로 진행을 멈추지 말고 임시 구성으로 먼저 완성해줘.
- 지금 단계에서는 멀티플레이, 저장, 인벤토리 같은 부가 시스템을 넣지 마.
- GUI 에디터를 열지 말고, 애셋 생성이 필요하면 커맨드렛으로 에디터 코드를 실행해서 처리해줘.
- 기존 사용자 변경사항이 있으면 되돌리지 마.

검증 요구사항:

- 구현 후 플레이어 스폰, 이동, 카메라 추적, 마우스 조준이 실제로 가능한지 확인해줘.
```

## 메모

- 이 문서는 단일 태스크용이 아니라 현재 CodexHarness 프로젝트 전체를 끌고 가기 위한 상위 하네스다.
- 실제 구현은 `한 번에 끝내는 방식`보다 `마일스톤 단위`로 나누는 편이 훨씬 안전하다.
- 첫 구현 목표는 반드시 `움직이는 플레이어`여야 한다. 적이나 UI부터 만들기 시작하면 구조가 뒤틀리기 쉽다.
