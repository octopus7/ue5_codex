# CodexPlanet 프로젝트 하네스

최종 수정일: 2026-04-04

## 문서 목적

이 문서는 `CodexPlanet`의 장기 작업을 같은 방향으로 이어가기 위한 최상위 기준 문서다.
프로젝트 목표, 범위, 구현 원칙, 마일스톤, 문서 운영 규칙을 고정한다.

장기 작업 중에는 아래 5개 문서를 함께 사용한다.
- `Docs/Harness/PROJECT_HARNESS_KO.md`
- `Docs/Harness/CURRENT_PHASE_KO.md`
- `Docs/Harness/DECISION_LOG_KO.md`
- `Docs/Harness/STATE_SNAPSHOT_KO.md`
- `Docs/Harness/WORK_TIME_LOG_KO.md`

## 프로젝트 요약

- 프로젝트명: `CodexPlanet`
- 엔진 버전: `UE 5.7`
- 대상 플랫폼: `PC`
- 네트워크 모델: `Single Player`
- 기본 맵: `/Game/Maps/BasicMap`
- 장르 목표: `행성 경영/배치 프로토타입`
- 핵심 상호작용: `드래그로 행성을 트랙볼처럼 회전`하고 `행성 표면에 프랍 배치`
- 배치 규칙 핵심: `행성`과 `행성 위에 놓인 오브젝트` 모두 배치 표면이 될 수 있음
- 행성 초기 표현: `코드 생성된 약간의 노이즈 스피어`
- 아트 방향: `행성은 코드 생성`, `프랍은 cute 스타일의 VOX`
- 현재 상태: 기본 레벨과 기본 런타임 모듈만 있는 거의 빈 프로젝트 상태

## 장기 목표

`CodexPlanet`의 1차 목표는 작은 행성 하나를 중심으로 관찰, 회전, 배치가 가능한 플레이어블 프로토타입을 만드는 것이다.

1차 완료 기준은 아래와 같다.
- 화면에 하나의 행성이 명확하게 보인다.
- 초기 행성은 코드로 생성한 약간의 노이즈가 있는 스피어로 표현된다.
- 사용자가 마우스 드래그로 행성을 `Google Earth`의 트랙볼처럼 자연스럽게 회전시킬 수 있다.
- 사용자가 행성 표면의 유효 위치를 가리키고 프랍을 배치할 수 있다.
- 배치되는 프랍은 행성 표면 노멀에 맞춰 방향과 위치가 정렬된다.
- 배치 가능한 프랍은 기존에 배치된 프랍 위에도 적층할 수 있다.
- 적층 시에도 새 프랍은 접촉한 표면 삼각형의 노멀 방향을 따라 정렬된다.
- 최소 5종의 귀여운 VOX 프랍이 실제 게임 애셋으로 들어와 배치 가능하다.
- 초기 프랍 후보는 `아치형 나무다리`, `석상`, `야자수`, `바위`, `수풀`이다.
- 행성 본체 시각 표현은 나중에 외부에서 가져온 메시로 교체할 수 있어야 한다.

초기 단계의 핵심은 `경영 수치`보다 `회전 + 표면 배치 + 귀여운 프랍 구성`을 먼저 플레이 가능한 수준으로 만드는 것이다.
경영 요소는 그 위에 얹는 후속 확장으로 본다.

## 전역 구현 원칙

- 이 프로젝트를 탑다운 슈터 전제로 해석하지 않는다.
- 플레이어 아바타 이동보다 `행성 관찰과 월드 조작`이 우선이다.
- 먼저 작은 수의 확실한 상호작용을 만든 뒤 시스템을 확장한다.
- 핵심 런타임 구조는 C++ 중심으로 두고, Blueprint는 연결, 배치, 애셋 바인딩, 튜닝에 사용한다.
- 없는 시스템을 있는 것처럼 문서화하지 않는다.
- 범위가 바뀌면 문서부터 갱신하고 구현은 그 다음에 진행한다.
- 행성 본체는 초기에 VOX를 사용하지 않는다.
- 행성은 코드 생성된 스피어를 기준으로 시작하되, 나중에 외부 메시로 갈아끼울 수 있는 구조여야 한다.
- 행성 상호작용과 표면 배치 로직은 특정 시각 메시 타입에 강하게 묶지 않는다.
- 배치 시스템은 `행성 표면`과 `배치된 오브젝트 표면`을 같은 개념의 배치 대상 표면으로 다룰 수 있어야 한다.
- 배치 정렬 기준은 접촉 지점의 `표면 삼각형 노멀`이다.
- VOX 프랍은 처음부터 실제 작업 대상 자산으로 취급한다.
- 아트는 사실적 디테일보다는 읽기 쉬운 실루엣과 귀여운 비율을 우선한다.

## 현재 기준 제약

- 이미 존재하는 `/Game/Maps/BasicMap`은 유지한다.
- 현재 확인된 런타임 모듈은 `CodexPlanet` 하나다.
- 에디터 자동화 모듈, 커맨드렛, 별도 애셋 파이프라인은 아직 없다.
- `EnhancedInput` 모듈은 의존성에 포함돼 있으며 기본 입력 시스템으로 사용한다.
- 현재 단계에서는 특정 저장 시스템, 경영 수치 시스템, 인벤토리 시스템을 선구축하지 않는다.
- 행성의 `고리 Niagara System` 또는 `위성`을 추가하는 UI는 반드시 에디터에서 확인 가능한 실존 `Widget Blueprint` 애셋으로 존재해야 한다.
- 해당 `WBP`의 위젯 트리는 에디터 코드로 작성되어 애셋으로 저장되어야 한다.
- 런타임 `NativeConstruct` 시점에 위젯 트리를 동적으로 조립하는 구현은 기본 경로로 사용하지 않는다.

## 하네스 보존 규칙

- `Docs/HarnessSeed`는 최초 기준 상태를 보존하는 고정 입력으로 취급한다.
- `Docs/Harness`는 실제 수행 중 계속 바뀌는 live 문서로 취급한다.
- `Docs/HarnessRuns`는 각 수행 결과를 보관하는 run 아카이브로 사용한다.
- 같은 초기 조건에서 재수행하려면 seed를 live harness로 다시 복원하는 방식을 사용한다.
- 재현성 비교는 가능한 한 `같은 seed`, `같은 코드베이스`, `같은 사용자 지시`를 기준으로 한다.

## 마일스톤 해석 규칙

- 이 문서의 마일스톤은 `CodexPlanet` 작업을 앞으로 어떻게 진행할지 정리한 기준선이다.
- `M0`, `M1`은 이번 문서 정리 과정에서 정의된 단계이며, 과거 구현 이력을 이어받았다는 뜻이 아니다.
- 현재 프로젝트는 실제 구현 기준으로는 `거의 비어 있는 상태에서 처음부터 시작`하는 프로젝트다.
- 따라서 이후 구현 작업은 사실상 첫 구현 단계부터 쌓아 올린다고 본다.
- 실제 첫 구현 단계 ID는 `M1-P0`부터 시작한다.

## 초기 구조 기준

### 런타임 구조 초안

- `GameMode`
- `PlayerController`
- 카메라 조작용 `Pawn` 또는 동등한 뷰 제어 액터
- 행성 자체를 담당하는 `PlanetActor`
- 행성 시각 표현을 분리하는 교체 가능 레이어
- 배치 가능한 프랍의 공통 베이스 액터 또는 데이터 구조

### 콘텐츠 폴더 초안

- `/Game/Core`
- `/Game/Blueprints`
- `/Game/Planets`
- `/Game/Props`
- `/Game/UI`
- `SourceArt/Vox/Props`

실제 폴더 세부 구조는 구현이 시작되면 `DECISION_LOG_KO.md`와 `STATE_SNAPSHOT_KO.md`에 확정 반영한다.

## 초기 VOX 프랍 기준

- 소스 자산은 기본적으로 `SourceArt/Vox/Props/` 아래에 둔다.
- 게임 애셋은 기본적으로 `/Game/Props/` 아래에 정리한다.
- 초기 제작 우선 프랍은 아래 5종이다.
  - `아치형 나무다리`
  - `석상`
  - `야자수`
  - `바위`
  - `수풀`
- 초기 목표는 프랍 수를 늘리는 것보다 `배치 품질과 시각 통일감`을 먼저 확보하는 것이다.

## 행성 표현 기준

- 행성 본체는 초기 단계에서 `.vox`로 만들지 않는다.
- 첫 구현은 코드로 생성한 `약간의 노이즈가 있는 스피어`를 사용한다.
- 행성 표면 판정, 배치 기준점, 노멀 계산은 가능한 한 `PlanetActor`의 공통 인터페이스 또는 공통 함수 레이어를 통해 제공한다.
- 이후 외부에서 제작한 메시로 교체하더라도 입력, 배치, 프리뷰, 프랍 시스템 전체를 다시 짜지 않도록 구조를 유지한다.

## 배치 규칙 기준

- 프랍은 행성 표면에만 놓는 것으로 제한하지 않는다.
- 이미 행성 위에 배치된 오브젝트도 다시 배치 가능한 표면이 될 수 있어야 한다.
- 즉 적층 배치를 지원하는 것을 기본 규칙으로 둔다.
- 새로 배치되는 오브젝트는 자신이 접촉하는 표면의 삼각형 노멀 방향을 따라 정렬된다.
- 이 규칙은 행성 표면이든, 행성 위에 올라간 오브젝트 표면이든 동일하게 적용한다.
- 표면 판정과 배치 정렬 로직은 특정 자산 종류가 아니라 `배치 가능 표면` 개념을 기준으로 설계한다.

## UI 애셋 기준

- 행성 주위를 도는 `고리 Niagara System`을 추가하거나 `위성`을 추가하는 조작은 전용 UI로 제공할 수 있어야 한다.
- 이 UI는 에디터에서 열어 구조를 직접 확인할 수 있는 실존 `WBP` 애셋이어야 한다.
- 위젯 트리는 에디터 코드로 만들어 `Widget Blueprint` 애셋 안에 저장하는 방식을 기본값으로 둔다.
- 런타임 `NativeConstruct`에서 위젯을 한 줄씩 만들어 조립하는 방식은 완료 상태로 간주하지 않는다.
- 즉 나중에 구현할 때도 `에디터에서 보이는 WBP 애셋 실존`이 검증 기준에 포함된다.

## 단계 운영 규칙

- 실제 작업은 반드시 `CURRENT_PHASE_KO.md`의 범위 안에서만 진행한다.
- 새 규칙, 예외, 구조 변경은 반드시 `DECISION_LOG_KO.md`에 기록한다.
- 코드, 설정, 콘텐츠 상태가 달라지면 `STATE_SNAPSHOT_KO.md`를 갱신한다.
- 각 단계의 시작 시각, 종료 시각, 소요 시간은 `WORK_TIME_LOG_KO.md`에 append 한다.
- 단계가 끝나면 `CURRENT_PHASE_KO.md`의 검증 결과와 다음 단계 후보를 갱신한다.

## 성공 조건

- 문서만 읽고도 다음 작업자가 현재 방향과 다음 구현 순서를 이해할 수 있어야 한다.
- `탑다운과 무관한 프로젝트`라는 점이 문서 전반에서 흔들리지 않아야 한다.
- 컨셉, 런타임 구조, 콘텐츠 방향, 현재 공백이 구분되어 보여야 한다.
- 실제 진행이 쌓일수록 문서가 작업 이력을 압축한 실행 기준으로 기능해야 한다.
