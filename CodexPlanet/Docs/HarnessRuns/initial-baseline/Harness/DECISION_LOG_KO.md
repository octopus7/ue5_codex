# CodexPlanet 결정 로그

최종 수정일: 2026-04-04

## 사용 규칙

- 구조, 범위, 규칙이 바뀌면 이 문서에 먼저 남긴다.
- 아직 구현되지 않은 항목이라도 이후 작업 방향에 영향을 주면 기록한다.
- 결정은 가능한 한 짧고 명확하게 적는다.

## 결정 사항

### 2026-04-04 | 문서 구조 | 5문서 하네스 구조를 `CodexPlanet`에 도입

- `PROJECT_HARNESS`, `CURRENT_PHASE`, `DECISION_LOG`, `STATE_SNAPSHOT`, `WORK_TIME_LOG` 구조를 기본 작업 프레임으로 사용한다.
- 이후 장기 작업은 이 문서 세트를 기준으로 이어간다.

### 2026-04-04 | 문서 기준 | 상위 하네스 내용은 복사하지 않고 뼈대만 차용

- `CodexHarness`에서 문서 역할과 운영 방식만 가져온다.
- 현재 프로젝트에 없는 시스템, 자동화, 애셋 파이프라인은 가져오지 않는다.

### 2026-04-04 | 프로젝트 정체성 | 기존 탑다운 전제는 전부 폐기

- 이 프로젝트는 탑다운 슈터와 무관하다.
- 기존 하네스에서 가져온 탑다운 관련 방향은 `CodexPlanet` 기준으로 사용하지 않는다.

### 2026-04-04 | 핵심 컨셉 | 단일 행성 경영/배치 프로토타입으로 고정

- 플레이 대상은 행성 하나다.
- 초기 핵심 경험은 행성을 관찰하고 회전시키며 표면에 프랍을 배치하는 것이다.
- 복잡한 경영 수치보다 관찰/회전/배치 루프를 먼저 만든다.

### 2026-04-04 | 주 입력 모델 | 드래그 기반 트랙볼 회전 사용

- 마우스 드래그로 행성을 `Google Earth`처럼 트랙볼 느낌으로 회전시키는 조작을 기본 상호작용으로 둔다.
- 플레이어 캐릭터 이동은 현재 핵심 입력 모델이 아니다.

### 2026-04-04 | 표면 상호작용 | 행성 표면에 사물 배치를 핵심 루프로 채택

- 사용자는 행성 표면의 위치를 선택해 프랍을 배치할 수 있어야 한다.
- 프랍은 행성 표면 노멀에 맞춰 정렬되는 방향을 기본값으로 둔다.

### 2026-04-04 | 적층 배치 | 행성과 배치 오브젝트를 모두 배치 표면으로 취급

- 프랍은 행성 표면뿐 아니라 이미 배치된 오브젝트 표면에도 놓을 수 있어야 한다.
- 즉 배치 시스템은 적층을 지원해야 한다.

### 2026-04-04 | 배치 정렬 기준 | 접촉 표면 삼각형의 노멀 방향을 따른다

- 새로 배치되는 오브젝트는 접촉한 표면의 삼각형 노멀 방향을 따라 정렬된다.
- 이 규칙은 행성이든 행성 위에 올라간 오브젝트든 동일하게 적용한다.

### 2026-04-04 | 행성 표현 | 행성 본체는 초기 단계에서 코드 생성 noisy sphere 사용

- 행성 자체는 `.vox`로 만들지 않는다.
- 초기 행성은 코드로 생성한 약간의 노이즈가 있는 스피어를 기본 표현으로 사용한다.

### 2026-04-04 | 교체 가능성 | 행성 시각 표현은 외부 메시로 교체 가능해야 함

- 현재는 코드 생성 스피어로 시작하더라도, 나중에는 외부에서 만들어온 메시로 교체할 수 있어야 한다.
- 입력, 배치, 표면 판정 로직이 특정 시각 메시 타입에 고정되지 않도록 구조를 나눈다.

### 2026-04-04 | 아트 방향 | 귀여운 VOX 프랍을 실제 작업 대상 자산으로 채택

- 프랍은 가능한 한 `.vox` 소스로 제작하고 게임 애셋으로 가져오는 방향을 기본값으로 둔다.
- 초기 우선 프랍은 `아치형 나무다리`, `석상`, `야자수`, `바위`, `수풀`이다.
- 스타일 우선순위는 사실성보다 `cute`, `readable silhouette`, `소형 행성에 어울리는 비율`이다.

### 2026-04-04 | 구조 원칙 | 플레이어 캐릭터보다 행성 뷰 제어 구조를 우선

- 초반 런타임 뼈대는 `GameMode`, `PlayerController`, 뷰 제어용 `Pawn`, `PlanetActor` 중심으로 잡는다.
- `Character` 중심 구조는 현재 기본 전제로 채택하지 않는다.

### 2026-04-04 | 문서 목적 | 진행 과정을 기준으로 장기 수행 작업을 지시하는 문서를 유지

- 이 문서 세트의 목적은 단순 기록이 아니라 다음 작업을 지시할 수 있는 실행 기준을 유지하는 것이다.
- 실제 진행이 생길 때마다 상태와 단계 문서를 갱신해 이어서 작업 가능해야 한다.

### 2026-04-04 | 마일스톤 해석 | 이전 구현을 이어받는 프로젝트처럼 적지 않음

- 현재 프로젝트는 거의 비어 있는 상태에서 처음부터 시작한다.
- 문서 안의 `M0`, `M1`은 과거 구현 히스토리가 아니라 이번 하네스 정리 과정에서 정의한 기준 단계다.
- 따라서 이후 실제 코드/콘텐츠 작업은 첫 구현부터 쌓아 올리는 흐름으로 기록한다.

### 2026-04-04 | 재현성 운영 | seed/live/archive 구조로 하네스 문서를 관리

- `Docs/HarnessSeed`는 고정 입력을 보존한다.
- `Docs/Harness`는 실제 수행 중 갱신되는 live 문서다.
- `Docs/HarnessRuns`는 run별 결과 아카이브다.
- 하네스를 다시 비우고 수행할 때는 seed에서 live로 복원하는 방식을 기본값으로 둔다.
