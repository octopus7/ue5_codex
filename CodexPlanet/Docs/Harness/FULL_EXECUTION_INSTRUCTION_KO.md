# CodexPlanet 전체 수행 지시문

최종 수정일: 2026-04-04

## 문서 목적

이 문서는 새 대화에서 `CodexPlanet`의 전체 계획을 이어서 수행할 때 그대로 사용할 수 있는 실행 지시문이다.
실제 작업 기준은 항상 `Docs/Harness`의 live 문서다.

## 최우선 규칙

- 작업 기준은 반드시 `Docs/Harness`다.
- `Docs/HarnessSeed`는 보존용 원본이므로 수행 중 수정, 삭제, 이동, 덮어쓰기, 포맷 변경 대상이 아니다.
- 수행 중 발생하는 문서 변경은 `Docs/Harness`의 live 문서에서만 반영한다.
- seed를 새 기준선으로 갱신하는 일은 일반 수행 흐름에 포함하지 않는다.
- seed 갱신이 필요하면 run 종료 후 별도 기준선 갱신 결정과 검토를 거쳐 의도적으로만 수행한다.
- `Docs/HarnessRuns`는 수행 결과 보관용 아카이브다.

## 시작 전 읽을 문서

- `Docs/Harness/README.md`
- `Docs/Harness/PROJECT_HARNESS_KO.md`
- `Docs/Harness/CURRENT_PHASE_KO.md`
- `Docs/Harness/DECISION_LOG_KO.md`
- `Docs/Harness/STATE_SNAPSHOT_KO.md`
- `Docs/Harness/WORK_TIME_LOG_KO.md`
- 이 문서 `Docs/Harness/FULL_EXECUTION_INSTRUCTION_KO.md`

## 시작 전 확인 절차

1. 현재 작업 기준이 `Docs/Harness`라는 점을 다시 확인한다.
2. live 하네스가 이전 run의 찌꺼기를 포함하고 있으면 먼저 정리 방법을 판단한다.
3. 새로 시작해야 하면 필요에 따라 `python Tools/harness/reset_harness.py --force` 또는 아카이브 후 reset 경로를 사용한다.
4. 기존 run을 보존한 뒤 초기화해야 하면 `python Tools/harness/reset_harness.py --archive-existing`를 우선 검토한다.
5. 현재 `CURRENT_PHASE_KO.md`의 단계 ID, 목표, 완료 기준, 제외 범위, 검증 체크리스트를 읽고 이번 루프 범위를 잠근다.

## 프로젝트 고정 조건

- 이 프로젝트를 탑다운 슈터로 해석하지 않는다.
- 목표는 `단일 행성 경영/배치 프로토타입`이다.
- 핵심 상호작용은 `드래그 기반 트랙볼 회전`과 `행성 표면 프랍 배치`다.
- 행성 본체는 초기 단계에서 `코드 생성된 약간의 노이즈 스피어`를 사용한다.
- 프랍은 `귀여운 VOX` 방향으로 제작하고 게임 애셋화한다.
- 배치 시스템은 `행성 표면`과 `배치된 오브젝트 표면`을 같은 `배치 가능 표면` 개념으로 다뤄야 한다.
- 배치 정렬 기준은 접촉 지점의 `표면 삼각형 노멀`이다.
- 프랍은 행성 표면뿐 아니라 이미 배치된 오브젝트 표면에도 적층 가능해야 한다.
- 핵심 런타임 구조는 C++ 중심으로 만들고 Blueprint는 연결, 배치, 애셋 바인딩, 튜닝에 사용한다.
- `/Game/Maps/BasicMap`은 유지한다.
- `고리 Niagara System` 또는 `위성` 추가 UI는 에디터에서 열어 확인 가능한 실존 `Widget Blueprint` 애셋이어야 한다.
- 런타임 `NativeConstruct`에서 위젯 트리를 조립하는 방식은 완료 상태로 간주하지 않는다.
- 멀티플레이, 저장/로드 선구축, 복잡한 경제 시뮬레이션 선구축은 범위 밖이다.

## 문서 운영 규칙

- 실제 구현은 항상 현재 `Docs/Harness/CURRENT_PHASE_KO.md` 범위 안에서만 진행한다.
- 새 규칙, 예외, 구조 변경, 단계 분해 방식 변경이 필요하면 먼저 `Docs/Harness/DECISION_LOG_KO.md`에 기록한다.
- 코드, 설정, 애셋, 폴더 구조, 문서 구조가 바뀌면 `Docs/Harness/STATE_SNAPSHOT_KO.md`를 갱신한다.
- phase가 끝나면 `Docs/Harness/CURRENT_PHASE_KO.md`와 `Docs/Harness/WORK_TIME_LOG_KO.md`를 함께 갱신한다.
- 없는 시스템을 있는 것처럼 문서화하지 않는다.
- 범위가 바뀌면 구현 전에 문서부터 갱신한다.

## 실행 원칙

- 탐색, 구현, 검증을 가능한 범위에서 병렬화해 전체 리드타임을 줄인다.
- 병렬 수행 여부 판단에서는 토큰 비용보다 전체 리드타임 단축을 우선한다.
- 필요하면 `generator`와 `evaluator`를 분리 운영할 수 있다.
- `generator`는 구현과 코드 생산을 맡고 `evaluator`는 리뷰와 리스크 및 검증 관점 평가를 맡는다.
- 두 역할은 직접 통신하지 않고 메인 에이전트가 결과와 피드백을 중계한다.
- 플레이어 캐릭터 이동보다 행성 관찰과 월드 조작을 우선한다.
- 행성 표현과 상호작용 책임은 나중에 외부 메시로 교체 가능한 구조로 유지한다.
- 표면 판정과 배치 정렬 로직은 특정 메시 타입에 강하게 묶지 않는다.

## 반복 수행 루프

1. 현재 `CURRENT_PHASE_KO.md`에서 이번 phase의 목표, 완료 기준, 제외 범위, 산출물, 검증 체크리스트를 읽고 범위를 확정한다.
2. 필요한 코드, 설정, 애셋, 폴더 구조를 실제 프로젝트에 구현한다.
3. 빌드, 실행, 에디터 확인, 수동 테스트 등 가능한 검증을 수행하고 문제를 직접 해결한다.
4. 변경 사항이 생기면 `STATE_SNAPSHOT_KO.md`를 갱신한다.
5. 새 구조 결정이나 예외가 생기면 `DECISION_LOG_KO.md`를 먼저 갱신한다.
6. phase 완료 시 `CURRENT_PHASE_KO.md`의 검증 결과와 다음 단계 후보를 갱신한다.
7. `WORK_TIME_LOG_KO.md`에 작업 ID, 작업명, 시작 시각, 종료 시각, 소요 시간, 병렬 수행 여부, 메모를 append 한다.
8. 다음 phase를 정의하고 같은 절차를 반복한다.
9. 전체 1차 성공 조건이 충족될 때까지 멈추지 않는다.

## 현재 단계 진행 기준

- 현재 시작 phase는 `M1-P0`이다.
- `M1-P0`: 행성 뷰와 코어 런타임 골격 추가
- `M1-P1`: `EnhancedInput` 기반 드래그 입력과 트랙볼 회전
- `M1-P2`: 행성 및 오브젝트 공통 표면 hit 판정, 프리뷰, 적층 배치
- `M1-P3`: 초기 5종 VOX 프랍 제작, import, 카탈로그 연결
- `M1-P4`: `고리 NS` 및 `위성` 추가용 실존 `WBP` 애셋 구조 설계 및 구현
- 이후 단계는 `PROJECT_HARNESS_KO.md`의 `M6`, `M7` 목표를 만족하도록 필요한 phase로 세분화해 이어서 진행한다.
- phase ID 체계를 더 세분화해야 하면 그 규칙을 `DECISION_LOG_KO.md`에 남긴다.

## 최종 성공 조건

- 화면에 하나의 행성이 명확하게 보인다.
- 코드 생성 noisy sphere 기반 초기 행성이 동작한다.
- 사용자가 드래그로 자연스럽게 트랙볼 회전을 할 수 있다.
- 사용자가 행성 표면의 유효 위치에 프랍을 배치할 수 있다.
- 프랍이 표면 노멀에 맞춰 정렬된다.
- 배치된 프랍 위에도 적층 배치할 수 있다.
- 최소 5종의 귀여운 VOX 프랍이 실제 게임 애셋으로 들어와 배치 가능하다.
- 행성 본체 시각 표현은 나중에 외부 메시로 교체 가능한 구조다.
- `고리 NS` 또는 `위성` 추가 UI가 에디터에서 확인 가능한 실존 `WBP` 애셋으로 존재한다.
- live 하네스 문서와 실제 프로젝트 상태가 일치한다.

## 중간 보고 방식

- 매 작업 루프마다 현재 phase, 이번에 구현한 것, 검증 결과, 문서 갱신 내용, 다음 행동을 짧고 명확하게 보고한다.
- 막히면 추상적 설명 대신 구체적인 원인과 해결 옵션을 제시한다.
- 가능한 한 가장 안전한 로컬 해결책을 기본값으로 선택한다.
- 외부 자산, 사용자 판단, 권한, 엔진 제약 때문에 더 진행할 수 없을 때만 멈춘다.

## run 종료 절차

1. 현재 live 문서가 실제 상태와 일치하는지 확인한다.
2. 필요하면 `python Tools/harness/archive_harness.py --run-id <run-id>`로 결과를 아카이브한다.
3. seed는 자동으로 갱신하지 않는다.
4. 새 기준선을 만들기로 명시적으로 결정된 경우에만 live 결과를 검토한 뒤 seed 갱신 작업을 별도로 수행한다.
